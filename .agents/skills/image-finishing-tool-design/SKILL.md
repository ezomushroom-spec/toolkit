# Purpose

この skill は、生成画像や既存画像に対して仕上げ処理を行うアプリを設計、レビュー、再構築するときに使う。

対象は、汎用ペイントアプリや自由編集ツールではなく、仕上げ品質を安定して再利用できる専用ツールである。

次のようなときに使う。

- 画像仕上げアプリの製品方針を決めたい
- recipe / preset 中心の構成を設計したい
- preview と final apply の一貫性を重視したい
- 将来の batch apply を見据えた構造にしたい
- GUI と処理エンジンの責務分離を確認したい

次のようなときは他の skill を優先する。

- 既存 UI の問題点監査だけをしたいときは `ui-audit`
- UI の視覚階層や構図を詰めたいときは `ui-design-direction`
- 技術方式を広く比較したいときは `pre-implementation-diagnosis`

この skill は、それらの一般 skill では弱くなりやすい「仕上げ品質」「レシピ再利用」「preview / final 一貫性」「batch 志向」を強く補正するために使う。

# Core Mindset

この種のアプリは、何でもできる編集ソフトとして広げるより、仕上げ工程の価値を抽出して、再利用可能で安定したワークフローへ落とすことを優先する。

目標は次ではない。

- 軽量な Photoshop 風ツールを作ること
- レイヤー中心の自由編集を先に広げること
- 見た目の派手さで使いやすさを代用すること

目標は次である。

- 仕上げ品質を安定して出せること
- レシピや preset として再利用できること
- preview と最終出力の挙動差を小さくすること
- 将来のバッチ処理へつながること

# Priority Order

明示的な指示がない限り、次の順で最適化する。

1. 出力品質と目的達成
2. 機能的で実務向きの GUI
3. 安定した動作

流行の技術、見た目の新しさ、広すぎる拡張性は、これらより上に置かない。

# Product Framing

このアプリは次のように捉える。

- レシピベースの仕上げツール
- 生成画像の仕上げに特化した専用ユーティリティ
- 繰り返し使える処理ワークフローツール
- GUI を備えたバッチ対応志向の仕上げエンジン

次のようには捉えない。

- 汎用ペイントアプリ
- 自由編集中心のエディタ
- 何でも追加できる編集基盤

# What to Prefer

次の提案を優先する。

- 出力品質を制御しやすく改善できる
- レシピ保存、読込、再利用ができる
- preview と final apply が同じ定義を共有する
- 将来の batch apply に広げやすい
- 処理エンジンと UI が分離されている
- 部分適用、マスク、条件分岐などへ無理なく拡張できる
- 画像の破壊や誤上書きを避けやすい

# What to Avoid

次は避ける。

- 汎用エディタ方向への不要な拡張
- UI イベントコードの中に処理本体を閉じ込める構成
- preview と final output が大きく乖離する設計
- 動作安定前の過度な最適化
- GPU 前提でしか成立しない初期設計
- 流行や見た目を目的適合より優先する提案

# Design Rules

設計では次を基本にする。

- 処理は順序付きステップとして表現する
- レシピは、有効/無効状態を持つ順序付きステップ群とパラメータ集合として持つ
- preview と final execution は同じレシピ定義を共有する
- 画像ファイルとレシピファイルは分ける
- アプリ設定とレシピ設定は分ける
- 単発調整より、再利用可能な仕上げ手順を優先する

# GUI Rules

この領域でよい GUI は次を満たす。

- レシピ構造が見える
- ステップの有効/無効が分かりやすい
- 強度やパラメータ調整がしやすい
- before / after 比較がしやすい
- 保存、複製、バッチ適用が分かりやすい
- 進捗とエラーが見える
- 誤って破壊的操作をしにくい

見た目の派手さより、ワークフロー理解を優先する。

# Technology Selection Guidance

技術候補は次で評価する。

- 高品質な仕上げ出力に向くか
- 順序付き処理ステップを素直に表現できるか
- preview / final の整合を保ちやすいか
- ローカルファイル中心の運用に合うか
- batch 処理に広げやすいか
- 将来の機能追加で破綻しにくいか

GPU を使う場合も、GPU は製品の本質ではなく高速化手段として扱う。
可能なら CPU fallback や段階導入を残す。

# Review Lens

提案や設計をレビューするときは、少なくとも次を確認する。

- 仕上げ品質の向上につながるか
- 専用ツールとしての性格を保てているか
- レシピを再利用しやすいか
- preview と final execution が揃っているか
- 将来の batch apply に耐えるか
- GUI が実務向きで、見た目だけに寄っていないか
