# Purpose

この skill は、複雑な不具合調査、原因切り分け、厳格なコードレビュー、最小限で安全な修正方針の整理に使う。

目的は単にコードを直すことではない。次を優先する。

1. 不具合の再現条件と原因候補を明確にする
2. 既存機能や意図した挙動を勝手に壊さない
3. 例外、リソース問題、誤操作リスクを先に見つける
4. 修正が必要でも、影響範囲を最小に抑える
5. 案件を分解できるときだけ調査用サブエージェントを使い、主エージェントが結論を統合する

# Relationship to existing skills

- `code-review` は広く使える単発のコードレビュー用。複雑な原因調査や調査分担の判断までは持たない。
- `plan-review` は実装前の計画レビュー用。コードや不具合の原因切り分けには使わない。
- この skill は、不具合調査と厳格レビューを主軸にし、必要時だけ調査を分解する。

# Use this skill when

- 不具合の原因を調査したい
- 落ちる、固まる、保存できない、重い、再現条件が不明
- 直近変更で壊れた箇所を切り分けたい
- UI、非同期、スレッド、ファイル入出力、例外処理、リソース使用量を重点的に見たい
- 複数の観点から調査したほうが安全
- 厳格なコードレビューをしたいが、単純な style review では足りない

# Do not use this skill when

- 1 か所だけの軽微な修正
- 型ミスや単純な null 参照など、原因がほぼ確定している
- 計画だけをレビューしたい
- すぐ実装に入るだけの依頼
- 同じファイルを複数エージェントが同時に編集すると衝突しやすく、分担メリットが薄い

# Core operating rules

## 1. Classify the case first

最初に案件を次のどれかへ分類する。

- 単純案件
  - 1 ファイルまたは 1 原因にほぼ絞れている
  - 並列化の得が少ない
  - この場合はサブエージェントを使わず、主エージェントだけで進める

- 分解可能案件
  - 原因候補が複数あり、独立に調査できる
  - この場合はサブエージェントの使用を検討する

- 競合注意案件
  - 複数エージェントが同じファイル群を触ると衝突しやすい
  - この場合は調査だけを分担し、編集は主エージェントが最後に統合する

- 境界案件
  - 入口や観点は複数あるが、読む先や直す先が最終的に同じファイル群へ収束する
  - この場合は分解可能に見えても、追加 subagent の利得が薄いなら主エージェントだけでまとめてよい

## 2. Use subagents only when justified

次の条件を 2 つ以上満たすときだけ、サブエージェントを使う。

- 原因候補が複数ある
- ログ確認、例外経路確認、差分確認、再現条件探索などに分けられる
- 巨大コードベースで探索コストが高い
- UI、バックエンド、I/O、非同期処理など観点が分かれている
- 主スレッドに中間ログを大量に流すと判断が鈍る
- まず調査を広く行い、その後に主エージェントが結論を出すほうが安全

次の条件では原則としてサブエージェントを使わない。

- 1 か所だけの軽微な修正
- 原因がほぼ確定している
- ほぼ同じファイルを同時に編集する必要がある
- 調査より編集競合リスクのほうが高い
- 入口は複数でも、結局同じ集約点を主担当が読むほうが早い

## 3. Keep subagents in investigation roles

サブエージェントは「調べる役」に寄せる。実装主体にしない。

必要に応じて、次のように役割を分ける。

- Reproduction Agent
  - 再現条件、入力条件、操作順、環境差分を調べる
- Log and Exception Agent
  - ログ、スタックトレース、例外経路、失敗分岐を調べる
- IO and Threading Agent
  - ファイル入出力、ロック、スレッド競合、非同期処理の危険箇所を調べる
- Resource Agent
  - メモリ、GPU、CPU、不要なコピー、解放漏れ、無駄な再計算を調べる
- UI Safety Agent
  - 誤操作時の安全性、操作導線、キャンセル不能処理、進捗表示不足を調べる
- Diff Agent
  - 直近変更や関連差分から破壊的変更の可能性を調べる

## 4. Main agent responsibilities

主エージェントは次を担当する。

- 依頼の理解
- サブエージェント投入の要否判断
- 各調査結果の整理
- 矛盾の解消
- 原因候補の優先順位付け
- 安全な修正方針の設計
- 必要時のみ最終修正の実行

サブエージェントの生ログや長大な思考過程は、そのまま最終回答に出さない。必ず要約して統合する。

# Review policy

レビューでは次の 5 項目を必ず確認する。

## 1. Bug and exception risk

- ファイル不存在
- 権限不足、パス不正、文字コード差異
- 例外の握り潰し
- スレッド終了待ち不足
- UI スレッド以外から UI を触っていないか
- 初期化前アクセス、解放後アクセス
- 想定外入力や空データ

## 2. Resource waste

- 不要な全件読み込み
- 大きな配列や画像の重複保持
- GPU テンソルやモデルの不要な常駐
- 毎回再生成される重いオブジェクト
- ループ内での無駄な変換や再計算

## 3. Maintainability and extensibility

- 関数が大きすぎないか
- UI、処理、状態管理が密結合になっていないか
- 設定値や定数が散らばっていないか
- 分岐追加で壊れやすい構造でないか

## 4. Readability and understandability

- 変数名と関数名が役割を表しているか
- コメントが不足または嘘になっていないか
- 似た処理が重複していないか
- 1 つの関数が複数責務を持ちすぎていないか

## 5. UI safety and usability

- 破壊的処理の確認があるか
- 長時間処理中の進捗表示やキャンセル手段があるか
- 入力不備時の表示が分かりやすいか
- 保存失敗時に内容が失われないか
- ウィンドウサイズや状態遷移で表示崩れしないか

# Fix policy

修正するときは次を守る。

- 元の機能を勝手に削除しない
- 見た目だけ整えて挙動を変えない
- 変更は最小限に抑える
- 原因に直接効く修正を優先する
- ついでの全面リファクタはしない
- 例外処理、リソース解放、入力検証は実害があるなら優先する
- 競合しやすい変更は主エージェントが最後にまとめて適用する

# Output contract

最終回答は原則として次の構成で返す。

## 1. レビュー結果の要約

- 評価
- 何が問題か
- なぜ危ないか
- 何をどう直すべきか

## 2. 詳細レビュー

5 つの観点ごとに次を書く。

- 問題になりそうな箇所
- なぜ問題なのか
- どう直すのが現実的か

## 3. 修正方針

- 最小修正で済む案
- 安全性を上げる追加案
- 今回は直さず監視でよい点

## 4. 修正後コード

ユーザーが修正を求めた場合のみ提示する。ファイルごとに、コピペできる形で出す。

# Execution flow

1. 依頼内容と対象コードを読む
2. 単純案件か、分解可能案件か、競合注意案件かを判定する
   - 再現未確定で `厳しめに見てほしい` 依頼なら、まず strict review 主軸で始め、必要な部分だけ diagnosis に切り出す
3. 分解可能案件なら、必要最小限のサブエージェント構成を決める
4. 各サブエージェントに独立した観点を持たせて調査させる
5. 結果を主エージェントが統合し、原因候補を順位付けする
6. 修正が必要なら最小変更で案を作る
7. ユーザー要求があれば完全な修正コードを提示する

# Important restraints

- サブエージェントは万能ではない。分担メリットが薄いなら使わない
- サブエージェント同士で同じファイルを好き勝手に編集させない
- 中間ログを大量に最終回答へ出さない
- 不明点は断定せず、可能性と根拠を分けて書く
- 実装を求められていない限り、レビューと改善提案を優先する
- 非エンジニアにも分かる表現を優先する
