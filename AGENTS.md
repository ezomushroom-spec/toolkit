# Workspace Rules

## 1. 設計判断

- 技術選定、構成提案、実装計画では、明示的な指示がない限り、アプリの本来目的の達成、機能的で実用的な GUI、安定した動作の順で優先する。
- 新しさ、流行、見た目の派手さは、それ自体を優先理由にしない。
- 過度に汎用化されたツールより、目的特化のツールを優先する。明示依頼がない限り、専用ツールを汎用エディタ方向へ広げない。
- 単発の手作業より、再利用、反復運用、バッチ適用しやすさを重視する。
- この workspace における「よい GUI」は、分かりやすく、正しく操作しやすく、誤操作しにくく、状態、進捗、エラーが見えやすく、反復実務に向いていることを指す。
- 見た目の良さは、機能の明快さと安定性を満たした後で評価する。
- 処理ロジックと UI ロジックは分離し、業務ロジックを UI コードへ閉じ込めない。
- 将来 UI を差し替えても処理系を再利用しやすい構造と、単体プレビューとバッチ実行で整合する状態管理を優先する。
- 技術、ライブラリ、構成案を比較するときは、本来目的への適合性、必要な GUI や運用フローへの適合性、安定性と保守性、将来拡張のしやすさ、見た目の自由度や流行性の順で評価する。
- 推奨案を出すときは、なぜ本来目的に合うのか、どのようなトレードオフがあるか、見送った案を採用しなかった理由を明示する。

## 2. 実装前の確認

- 既存機能を削除しない。停止や置換が必要でも段階的に扱う。
- 実装前に対象画面、関連処理、データ影響、戻し方を整理する。
- 正本の判断は docs や README だけで決めず、実コード、実データ、現用設定を優先して確認する。
- 正本候補が複数ある場合は、読む先、実装先、文書更新先を一本化するまで実装しない。
- 中規模以上の案件では、実装前に進行フローを明示する。形式はフローチャート、段階表、依存整理のいずれでもよいが、少なくとも対象、実装順、確認点、戻し方を含める。
- 進行フローは調査→論点分解→最小実装単位の確定→実装→確認の順を基本にする。
- 長期化しやすい案件では、一度に広げすぎず、区切った単位で完了条件を置く。
- 長期化しやすい案件では、未決事項を減らしたら採用案を文書化して前進する。調査そのものを目的化しない。
- 小規模修正では、必要最小限の整理に留める。
- 新規 UI、技術選定、アーキテクチャ変更、クロスレイヤー設計を伴う案件では、実装前に実装方式の診断と比較を行う。
- 実装前診断では、可能なら 2〜3 の現実的な候補を比較し、無理に弱い案を増やさない。
- 実装前診断は、要件適合性、実装の確実性、保守性、拡張性、UI 品質、非専門ユーザーの運用しやすさ、既存資産の再利用を基準に行う。
- 既存の Python が補助スクリプトではなく業務ロジックや安定した処理資産として存在する場合は、backend または core として維持する案を必ず比較対象に含める。
- UI 品質や運用性が論点でも、構成分離は常に採用せず、複数画面、状態管理、配布形態、責務分離の必要性がある場合に限って frontend/backend 分離を明示比較する。
- 実装前診断の説明は、非専門ユーザーでも判断できる平易な日本語で行い、推奨案だけでなく見送る案の理由も示す。
- 実装前の調査、方式比較、仕様化、計画化、文書整理は、破壊的変更や重要トレードオフがない限りユーザーへの逐一確認なしで進めてよい。
- 診断の結果、重要なトレードオフが残る場合は実装前に確認を入れる。重要な分岐がない場合は、診断提示後に推奨案で進めてよい。
- 初期方針が与えられていても、体感速度、完成度、移行目的、正本の所在が強く衝突する場合は、その前提が固定条件か暫定条件かを途中で再確認する。
- `移行` は目的ではなく手段として扱い、旧版より重要な UX や速度要件が満たせないと判明した時点で、再構築案を比較対象へ戻す。
- `正本を残す`、`shell に留める`、`既存 UI を踏襲する` といった方針は、固定条件なのか再交渉可能な条件なのかを実装前に確認し、詰まりが出た時点でも問い直す。
- preview 主導やレビュー主導のツールでは、見た目の移植より体感速度と操作ループを優先論点として先に確認する。
- 中盤以降に最適化が連続しても UX が実用水準へ届かない場合は、微修正の継続ではなく「構造限界の可能性」「移行継続の妥当性」「再設計へ切る条件」を明示的に整理する。
- 案件開始時と大きな詰まり発生時には、少なくとも `本当に守るべきものは既存資産か UX か`、`段階移行で満たしたい成功条件は何か` を確認する。
- 小規模な不具合修正、局所的な文言修正、既存構成内の限定変更では、完全な方式比較を省略し、必要最小限の根拠整理に留める。
- 複雑案件では、実装前に `docs/templates/implementation-plan-template.md` 相当の計画を出す。

## 3. 実装中の原則

- 既存挙動の保全を優先し、無関係なリファクタ、命名整理、構成変更を勝手に混ぜない。
- UI改善は構造→見た目→安全性の順で行う。
- 主操作は最も見つけやすい位置と文言で示す。
- 危険操作には確認、対象明示、結果説明を入れる。
- ローディング中は二重実行を防ぎ、進行中であることを示す。
- エラーは次の行動が分かる形で表示する。
- 大きな変更は理由と影響を説明できる形で進める。
- 既存の設定 key、列名、preset format、保存ファイル形式を変える場合は、互換維持策か移行手順と戻し方を先に示す。
- 複数入口を持つツールでは、入口ごとの責務を混同せず、同じ業務ロジックを別入口へ重複実装しない。
- 既存のローカル Web アプリを Electron / Tauri へ広げる場合は、業務処理の正本を backend 側に残し、shell 側へ責務を複製しない。
- Electron / Tauri 併設時は、バックアップ、single instance、desktop 専用起動口、backend log、native picker/D&D、戻し方を最初に整理する。
- 通常ブラウザと Electron で D&D の成功率や取得可能情報は違う前提で設計する。

## 4. 品質確認

- 動いたように見えるだけでは完了扱いにしない。
- テストは「書いたこと」ではなく「実行可能で、変更の確認に使えること」を重視する。自動化できない場合も、再現可能な確認手順を残す。
- 確認対象には正常系だけでなく、失敗系、空入力、不正設定、存在しないパス、権限不足、途中停止やキャンセル後の状態を含める。
- ファイル削除、上書き、外部送信、ジョブ実行など誤操作コストの高い操作は、確認、対象明示、取り消し不能性の説明、実行中の二重防止を優先する。
- メモリ、GPU、長時間ジョブ、バックグラウンド常駐を伴う変更では、無駄な資源消費や解放漏れを確認対象に含める。
- 実装後レビューでは、少なくとも挙動差分、危険操作、二重実行防止、エラー表示、未考慮ケースを確認する。
- workspace 共通の品質基準は `docs/quality-gates.md` に置く。完了判定、確認観点、完了報告の詳細はこれを正本とする。
- 秘密情報、個人情報、実データ、外部送信の取り扱いは project 文書で許可条件と回避策を先に定義する。環境変数は `.env.example` のようなテンプレートでコミットし、`.env` 実体をエージェントに直接操作・出力させない。

## 5. 文書と配置

- project 個別の build/test/run 手順や優先確認項目は `projects/<name>/AGENTS.md` を基本とし、初見導線や要約は `START_HERE.md` や `PROJECT_SUMMARY.md` に分けてよい。
- バックアップ、生成物、`build`、`dist`、`*_profile`、`src_backup_*` は正本として扱わず、明示依頼なしで編集しない。
- `projects` には原則として active な案件と、その判断・計画・入口文書を置く。backup や停止済み案件を残す場合は、active と混同しない名前・文書・置き場を先に整理する。
- `archive` へ移すのは、active ではなくなった案件、再利用予定の低い退避コピー、完了済みで参照中心になった文書群を基本とする。移動前に旧→新の対応表と戻し方を残す。
- 調査で抽出した要旨、整理メモ、実装準備文書は元フォルダ直下に置かず、`E:\codex\workspace\projects` 配下に案件ごとのフォルダを作って配置する。
- `projects` 配下に残す調査・計画・レビュー文書は、少なくとも「現状確認」「採用判断」「未決事項」を分けて記録する。
- project 固有の制約や運用ルールは workspace 共通ルールへ混ぜず、project 配下の `AGENTS.md` か案件文書へ分離する。

## 6. ユーザー確認と委任

- ユーザー確認は、高リスクな破壊的操作、取り返しのつきにくい外部送信、コストの大きい方針転換、要件を変える判断に限定し、通常の実装準備や低リスクな文書更新では止まらない。
- サブエージェントは、ユーザーが委任を明示的に許可した場合にだけ使う。
- `必要なら深く見て`、`詳しく調べて` のような表現だけでは、main agent の深掘り許可とは読めても、subagent 利用許可とはみなさない。
- サブエージェントに渡す作業は、探索、限定実装、検証など境界が明確で非重複なものに絞る。
- 次の一手そのものが結果待ちになる作業は、安易にサブエージェントへ丸投げしない。
- サブエージェントを起動する際は、必要に応じて案件に即した視点や属性を明示する。  
  例: 利用者視点、運用者視点、二次創作作家視点、イラストレーター視点、配布保守視点。
- 実利用に近い判断が必要な案件では、コード観点だけでなく、少なくとも1つは利用者や運用者の視点を持つサブエージェントを含める。
- サブエージェントのモデルは、原則として `gpt-5.4-mini` を使う。
- 探索、要約、比較、読み取り中心の切り分け、限定スコープの軽微な実装や検証は、まず `gpt-5.4-mini` を第一候補にする。
- サブエージェントの推論労力は、まず `low` か `medium` を検討する。
- `gpt-5.4` 以上をサブエージェントへ使うのは例外扱いとし、複雑な設計判断、難しいデバッグ、再現困難な不具合、複数原因の競合、複数結果の統合判断、失敗コストが高い変更に限る。
- 迷った場合は `gpt-5.4-mini` を既定にし、何となく品質が不安という理由だけで上位モデルを既定にしない。
- 読み取り中心の切り分け、要約、比較、個別観点の調査では、安易に上位モデルへ上げない。
- 同じファイル群を複数サブエージェントで無秩序に並列編集しない。
- サブエージェントの結果は参考とし、最終判断と実装はメインが行う。
- サブエージェントの進捗共有と結果要約は、日本語で読める形にそろえる。

## 7. 変更後の確認

- 変更後は影響範囲を確認し、必要な動作確認を行う。

## 8. 変更の出口

- 変更を完了扱いにするときは、原則として `branch -> commit -> push -> PR` の順で外部反映する。
- 外部への反映はこれを標準処理とし、例外がある場合はその理由と戻し方を project 側の文書に明記する。
- AIエージェントが自律的にコミットを作成する場合は、`[AI] Refactor ...` のように明確なプレフィックスをコミットメッセージに付与し、人間側からのレビュー見通しを確保する。
