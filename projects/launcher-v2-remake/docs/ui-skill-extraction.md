# UI Skill Extraction

## Summary

この案件からは、Windows向けランチャーやツール管理UIに再利用しやすい判断基準をスキルとして抽出できる。

特に有用なのは、次の3系統。

- ランチャー一覧の可視性最適化
- タブ/ナビゲーションの面積圧縮
- 操作ボタンと情報密度の優先順位付け

## Candidate Skill 1

### Name

`windows-launcher-ui-density`

### Purpose

ランチャー系UIで「何が登録されているか」を一目で把握できるようにしつつ、1画面で扱える件数を増やす。

### Reusable rules

- 一覧の主役は操作ボタンではなくアプリ名
- カードは詳細説明より識別性を優先する
- フルパスは常時表示せず、短縮表示か補助情報へ落とす
- 件数が多い画面では縦積みよりグリッドを優先する
- ボタンは必要機能を残しつつ最小サイズへ圧縮する
- 7件前後をスクロールなしで把握できる密度を基準にする

### Trigger examples

- ランチャーUIが縦に長くて見づらい
- アプリが増えると一覧性が落ちる
- 1ページで登録項目を見渡したい

## Candidate Skill 2

### Name

`compact-tabbed-workspace-ui`

### Purpose

大タブ/小タブのような階層ナビゲーションを、内容領域を圧迫しない形で整理する。

### Reusable rules

- ナビゲーションは内容より目立ってはいけない
- 左サイドに分厚く置くより、上部の細い帯で済むならそちらを優先する
- タブ階層が2段までなら、上部2段バー構成をまず検討する
- 編集操作付きタブでも、内容領域を削りすぎない幅に収める
- 内容一覧の主要ユースケースが「見渡すこと」の場合、ナビの常設面積は小さくする

### Trigger examples

- タブUIが大きすぎて一覧領域を圧迫している
- エクスプローラ風ナビと従来タブのどちらがよいか迷う
- 階層ナビを保ったまま一覧表示を広げたい

## Candidate Skill 3

### Name

`safe-density-first-ui-remake`

### Purpose

既存機能を落とさずにUI密度と視認性を改善するための改修順を固定する。

### Reusable rules

- 先に構造を見直す
- 次に情報優先度を整理する
- 最後に見た目を詰める
- 既存操作は削除せず、まず縮約と配置変更で対処する
- 危険操作は小さくしても隠さない
- 比較対象となる旧UIがある場合は、利便性の強みを先に言語化してから新UIへ反映する

### Trigger examples

- 既存UIの方が便利だと感じる
- 新UIが広いのに見づらい
- 改修で情報密度と安全性の両立が必要

## Best fit for this project

この案件に最も合う単独スキルは `compact-tabbed-workspace-ui` と `windows-launcher-ui-density` の組み合わせ。

もし1本にまとめるなら、次のような統合スキルにできる。

### Proposed integrated skill

`launcher-ui-remake-safe`

### Scope

- Windows向けランチャー
- タブ/カテゴリ/グループ付き一覧UI
- ショートカットやツール集の管理画面
- 「一覧性」「1画面収まり」「既存操作維持」が重要な案件

### Core rules

- タブは内容領域を食いすぎない
- 一覧はアプリ名優先で見せる
- パスや補足は二次情報として圧縮する
- 操作は残すが、情報認知を邪魔しないサイズにする
- 従来UIの利便性が高い場合は、その構造的強みを継承する
- UI改善は構造 -> 情報優先度 -> 見た目 -> 安全性確認の順で進める
