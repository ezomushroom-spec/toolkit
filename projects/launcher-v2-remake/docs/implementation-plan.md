## implementation-planner

### 1. Goal

現行 `ランチャー_v2` の主要挙動を安全に再現できる最小再開発範囲を定め、設定互換と操作互換を崩さない実装順を固定する。

### 2. Target files

- 現行参照元
  - `E:\自作アプリ集\ランチャー_v2\main.py`: 起動フロー全体
  - `E:\自作アプリ集\ランチャー_v2\src\ui\app.py`: 主要UIと操作導線
  - `E:\自作アプリ集\ランチャー_v2\src\core\config_manager.py`: 永続化と状態管理
  - `E:\自作アプリ集\ランチャー_v2\src\core\shortcut_launcher.py`: 実行責務
  - `E:\自作アプリ集\ランチャー_v2\src\services\hotkey_service.py`: グローバルホットキー
  - `E:\自作アプリ集\ランチャー_v2\src\services\tray_service.py`: トレイ常駐
- 再開発先候補
  - `E:\codex\workspace\projects\launcher-v2-remake\app\`
  - `E:\codex\workspace\projects\launcher-v2-remake\docs\`

### 3. Ordered implementation steps

1. 設定スキーマとモデルの固定
   - Affected files: `app/config/*`, `app/models/*`, `docs/*`
   - Why now: すべての UI と操作がこの定義に依存するため
2. 起動/保存/終了のコア責務を分離
   - Affected files: `app/main*`, `app/core/config*`, `app/core/launcher*`, `app/services/*`
   - Why now: 常駐アプリとしての成立条件を先に満たすため
3. タブ・グループ・ショートカット編集 UI を最小移植
   - Affected files: `app/ui/*`
   - Why now: 現行データを触れる状態を早く作るため
4. 検索候補、最近使用、ピン留め、表示モードを接続
   - Affected files: `app/ui/*`, `app/core/config*`
   - Why now: 日常利用の主要導線を keep 完了に近づけるため
5. 安全性と不一致修正
   - Affected files: `app/ui/*`, `app/services/*`
   - Why now: ダミー機能、二重実行、エラー導線をここで整えるため
6. 後回し機能を段階導入
   - Affected files: アイコン、検索フィルタ、個別ホットキー関連
   - Why now: keep を崩さず改善を積むため

### 4. Risks

- 設定 JSON のネスト構造を変えると移行コストが一気に上がる
- 絶対パス前提のショートカットが多く、環境差で起動失敗が出やすい
- ホットキーと GUI スレッドの連携を誤ると表示不整合やフリーズが起きうる
- トレイ終了とウィンドウ close の責務差を崩すとプロセス残留しやすい
- Windows API 依存を薄めすぎると現行の運用性を落とす恐れがある

### 5. Unknowns requiring final judgment

- 再開発先の UI 技術選定
- 設定ファイルの完全互換範囲
- 「管理者として実行」を keep に含めるか later に送るか
- 実アイコン表示を初回から含めるか

### 6. Suggested implementation scope

初回は `settings.json` 互換を維持したまま、通常起動、トレイ、ホットキー、階層編集、D&D追加、検索候補起動、最近使用、ピン留め、表示モード保存までを最小安全スコープとする。

## コードレビュー観点での再確認

- `ConfigManager` の二重生成は、再開発時に必ず単一責務へ寄せた方がよい
- ダミーの「管理者として実行」は、残すなら未実装表示、実装しないなら一時的に隠す判断が必要
- README と実装の差分は、再開発開始時点で仕様書側に吸収しておかないと期待値がずれる
- 設定インポートは全上書きのため、初回再開発ではバックアップ作成を必須導線にした方が安全
