## 3. 論点分解

- UI の論点:
  - 履歴クリアの結果が見た目だけで終わらないようにする
  - エラーや確認文言を利用者に分かる形で出す
- データ / 設定の論点:
  - `requirements.txt` に残っている `mediapipe` の扱い
  - 履歴保存ファイルとの整合
- 外部 I/O の論点:
  - 画像読み込み時の EXIF 回転
  - JPEG 書き出し結果とプレビューの一致
- 互換性の論点:
  - 既存プロジェクト保存データとの読み込み互換
  - 既存の一括出力導線を消さずに挙動だけ整える
- 運用上の論点:
  - 今後 `mediapipe` を復活させる予定があるか
  - 一括出力を「現在調整中の結果を基準にする」のか「各プリセット既定値を使う」のか

## 5. 確認計画

- 正常系:
  - 画像追加 -> プリセット選択 -> プレビュー更新 -> 単体出力
  - 同条件で全プリセット出力
  - 履歴表示 -> 履歴クリア -> 再起動
- 失敗系:
  - 存在しない画像パスを含むプロジェクト読込
  - 書き出し先が使えない場合のエラー
- 空入力:
  - 画像未選択で単体出力 / 一括出力
  - 履歴 0 件で履歴操作
- 不正設定:
  - プロジェクト JSON に古い `crop_offsets` だけが残る場合
- 存在しないパス:
  - 保存済み画像が消えた状態で履歴選択
- 途中停止 / キャンセル:
  - ダイアログをキャンセルした場合に状態が壊れないか
- 危険操作:
  - 履歴クリア
  - 出力済みファイルの上書き導線
- メモリ / GPU / 長時間処理:
  - 大きい画像の読み込みで極端なメモリ増加がないか
  - プレビュー更新の連打で UI が固まらないか
