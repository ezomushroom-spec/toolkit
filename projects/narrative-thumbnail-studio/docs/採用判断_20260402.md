# 採用判断

## 実装前診断

### 1. 要求の整理

- 依頼の本質:
  - このアプリの現行スタックが用途に対して妥当かを見極め、今後の改善方針を決めやすくすること
- 技術的に重要な論点:
  - GUI の作りやすさ
  - 画像合成処理との近さ
  - ローカル運用のしやすさ
  - 余計な依存を抱えていないか
- 既存資産の扱い:
  - `core/` の画像処理ロジックと `ui/` の PySide6 画面は既存資産として再利用価値が高い
- 既存処理やデータへの影響:
  - 出力履歴、プリセット JSON、起動バッチ、画像プールの導線に影響する

### 2. 候補アプローチ

#### 候補A

- 概要:
  - `Python + PySide6 + Pillow + NumPy/OpenCV` を維持し、不要依存だけ整理する
- 強み:
  - 既存資産をそのまま使える
  - 画像処理と UI の距離が近く、修正が速い
  - ローカル配布や単体実行の前提に合っている
- 弱み:
  - UI の洗練度や部品再利用は Web 系よりやや弱い
  - 依存管理を放置すると重くなりやすい
- 今回の適合度:
  - 最も高い

#### 候補B

- 概要:
  - Python コアを残しつつ、UI だけ別のデスクトップ shell へ広げる
- 強み:
  - UI の自由度を上げやすい
  - 将来的な画面刷新の余地を作れる
- 弱み:
  - 現在の規模では構成が過剰になりやすい
  - プレビュー操作や状態同期の責務が二重化しやすい
- 今回の適合度:
  - 今すぐは中程度

#### 候補C

- 概要:
  - Electron や Web UI ベースへ全面移行する
- 強み:
  - 見た目の自由度は高い
  - Web 系 UI 資産を使いやすい
- 弱み:
  - ローカル画像処理ツールとしては構成が重い
  - 既存 Python ロジックとの橋渡しが増え、保守箇所が増える
  - 今回の課題はスタック変更より挙動整合のほうが大きい
- 今回の適合度:
  - 低い

### 3. 推奨案

- 推奨する構成:
  - 候補A。現行の Python 中心構成を維持しつつ、不要依存を整理する
- 推奨理由:
  - このツールの本質はローカル画像処理とデスクトップ操作であり、現行構成はその要求に素直に合っている
  - 問題の中心はフレームワーク選定ではなく、出力仕様の不一致と依存の掃除不足にある
  - 既存の `core/` と `ui/` を活かしたほうが、実装の確実性と保守性が高い
- 見送った候補とその理由:
  - 候補Bは将来の画面刷新には有力だが、現時点の課題に対しては構成追加の負担が先に立つ
  - 候補Cは再構築コストに対し得られる利益が小さく、現行問題の解決策として遠回り
- 主なリスク:
  - `mediapipe` のような不要依存が残ると、起動環境や配布時の重さが無駄に増える
  - スタックを維持しても、挙動差分を放置すると利用者の不信感が残る
- 条件次第で有力になる次点案:
  - 画面数が増え、複数ワークフローや配布形態の整理が必要になった時は候補Bを再評価してよい

## 採用判断

- 採用:
  - 現行スタックを維持
- 直近でやるべきこと:
  - `mediapipe` 依存の実利用有無を確認し、不要なら削除候補にする
  - 出力系の仕様を単体/一括で揃える
  - 読み込み時の向き補正など、画像ツールとしての基本品質を先に上げる
- 今回やらないこと:
  - Web 化
  - Electron 化
  - UI フレームワークの全面入れ替え
