## 1. 確定事項

### 1.1 目的

Pixiv 投稿時のタグ入力を支援し、手入力負荷を下げつつ、タグ付けの再現性を高める。

本機能は候補提示・再利用・履歴利用によってユーザー判断を補助するものであり、タグの自動確定は行わない。

### 1.2 適用範囲

本機能は Pixiv のみを対象とする。

行うこと:

- Pixivタグ候補の提示
- 過去タグの再利用支援
- 同梱辞書に基づく候補提示
- 将来的な履歴学習のための基盤整備

行わないこと:

- Patreon 用タグ支援
- Discord 用タグ支援
- 他プラットフォーム向けラベル生成
- タグの自動確定
- 未確認タグの自動保存

### 1.3 `tags` の意味

現行 CSV 列 `tags` は、以後 `Pixiv 用タグ` として扱う。

- 永続列名は当面 `tags` のままとする
- UI 表記は `Pixivタグ` とする
- backend ロジックでも `Pixiv 用タグ` として扱う
- Patreon / Discord の業務仕様としては `tags` を用いない

将来的に必要であれば `tags_pixiv` へ移行できる設計とするが、初期導入では列名変更を行わない。

### 1.4 責務分離

- タグ支援ロジックは Python backend 側に置く
- Web UI / Electron UI は API を通じて利用する
- Electron 側へ辞書ロジックや学習ロジックを持ち込まない
- Pixiv セクション専用機能として扱い、Patreon / Discord UI へ波及させない

### 1.5 辞書の正本方針

Pixivタグ支援は、巨大な外部タグDBを正本にしない。

- 正本はローカル運用由来のタグ履歴とする
- `posts.csv` は投稿タスクの業務データ専用とし、辞書本体や学習結果を混在させない
- 同梱 seed はコールドスタート対策の補助辞書として扱う
- 将来の学習結果は `posts.csv` とは別保存にする

優先順位は以下とする。

1. ユーザー運用で確定保存された Pixivタグ
2. `posts.csv` から抽出した既存 Pixivタグ
3. 同梱 seed タグ辞書
4. 同梱 seed 作品辞書
5. 手動インポート辞書

外部サイトからの大規模タグ収集や、利用条件が不明な辞書の自動吸収は初期方針として採用しない。

手動インポート辞書を将来導入する場合は、以下を前提とする。

- ローカル限定で利用する
- 出所を明示できる
- 無効化できる
- 再配布前提の正本にはしない

### 1.6 二次創作向けの優先方針

二次創作向けの候補提示では、単純に `作品タグ > キャラタグ > 属性タグ` とせず、まず `外せない識別タグ` を優先する。

- 作品タグと主要キャラタグは、発見性に直結するため自動候補の主軸にしてよい
- カップリング、属性、解釈差が出やすいタグは誤候補のダメージが大きいため、初期段階では強く前に出しすぎない
- 同梱 seed は `作品の基本タグ` と `主要キャラ` を中心にし、汎用属性を増やしすぎない
- 学習結果は有用だが、仮タグや一時タグで汚れやすいため、自動確定には使わない

要するに、`外せないタグだけ先に出し、解釈が割れるタグは出しすぎない` ことを二次創作向けの基本方針とする。

### 1.7 タグ層と辞書担当の方針

Pixivタグ支援では、タグを少なくとも以下の層に分けて扱う。

1. 年齢区分
2. 作品タグ
3. 主要キャラタグ
4. 行為名 / シチュエーション
5. 性癖 / 解釈差が出やすい内容タグ

このとき、優先順位は popularity ではなく、`外すと誤解や事故が起きやすい層` を優先する。

- `年齢区分` は漏れコストが高いため、少数の明確なタグに限って同梱 seed に含めてよい
- `作品タグ` と `主要キャラタグ` は発見性に直結するため、同梱 seed と作品辞書の主軸にしてよい
- `行為名 / シチュエーション` は重要だが文脈依存が強いため、原則として learning / history 寄りで扱う
- `性癖 / 解釈差が出やすい内容タグ` は誤爆コストが高いため、seed で広く前に出しすぎない

要するに、`seed は作品 / キャラ / 少数の年齢区分`、`learning / history は行為 / シチュ / 性癖寄り` を基本とする。

## 3. Phase 分割

## Phase 1
実装対象を最小に絞る。

実装する:

- `tags` を UI 上で `Pixivタグ` と表示する
- 同梱 seed 辞書を読み込む
- `posts.csv` の既存 `tags` から頻出タグを抽出する
- タスク編集モーダルの Pixiv セクションに候補タグ UI を出す
- 候補クリックで入力欄へ追記する
- 同一タグは重複追加しない

実装しない:

- 学習 DB
- SQLite
- 共起学習
- タイトル語学習
- フォルダ名学習
- 作品辞書の高度順位補正

完了条件:

- 既存タスク編集を壊さず候補表示できる
- `tags` 保存互換を維持できる
- Pixiv 以外の UI に変化がない
- 候補機能が壊れても手入力で継続できる

## 4. Phase 1 のデータ仕様

### 4.1 Seed タグ辞書

保存先:

- `app/data/pixiv_seed_tags.json`

役割:

- 汎用的によく使う Pixiv タグの初期候補

想定形式:

```json
[
  { "tag": "オリジナル", "aliases": [], "category": "genre" },
  { "tag": "女の子", "aliases": [], "category": "character" },
  { "tag": "R-18", "aliases": ["成人向け"], "category": "rating" }
]
```

### 4.2 Seed 作品辞書

保存先:

- `app/data/pixiv_seed_works.json`

Phase 1 では軽量に導入し、順位補正よりも候補補助を主目的とする。

想定形式:

```json
[
  {
    "id": "blue_archive",
    "name": "ブルーアーカイブ",
    "aliases": ["ブルアカ", "Blue Archive"],
    "series": "ブルーアーカイブ",
    "characters": ["シロコ", "ホシノ"],
    "related_tags": ["ブルーアーカイブ", "ブルアカ"]
  }
]
```

### 4.3 過去 tags 抽出

`posts.csv` の `tags` を走査し、スペース区切りで分割して頻度集計する。

Phase 1 では以下に限定する。

- 単純頻度
- 重複除去
- 空文字除去

この段階では共起学習は行わない。

## 5. Phase 1 の候補生成仕様

候補生成元は以下の 3 系統に限定する。

1. 同梱 seed タグ辞書
2. `posts.csv` から抽出した既存 Pixivタグ
3. 軽量な作品辞書一致

優先順位は以下とする。

1. 過去 `posts.csv` 由来の頻出タグ
2. 作品辞書由来候補
3. seed タグ辞書由来候補

Phase 1 では、タイトル語やフォルダ名の自由解析を主軸にしない。
必要最小限の一致だけに留める。

## 6. UI 仕様

### 6.1 表示位置

- `Pixivタグ` 入力欄は Pixiv セクションに置く
- 既存の基本情報欄から即時に完全移設するかは実装時に選べるが、少なくとも UI 文言は `Pixivタグ` に統一する
- 候補 UI は Pixiv セクション配下へ表示する

### 6.2 追加 UI 要素

Phase 1 で追加してよい UI:

- 候補タグチップ群
- よく使う候補
- 作品候補
- 候補の出所を弱く示す補助表示
  - 例: `履歴`, `作品`, `同梱`, `学習`
  - ただし、タグ本体より目立たせず、判断補助に留める

将来的に追加してよい UI:

- `年齢区分`
- `作品 / キャラ`
- `内容候補`
  のような意味カテゴリ別の緩いグルーピング
- `もっと見る` や折りたたみで補助カテゴリを後段表示する構造

Phase 1 で追加しない UI:

- 複雑な辞書管理画面
- 学習管理画面
- 自動タグ修正 UI

### 6.3 操作仕様

- 候補クリックで現在の入力へ追記
- 同一タグがすでに存在する場合は追加しない
- 入力内容の削除と並び替えはユーザー手動
- 候補機能は補助であり、入力欄内容を自動書き換えしない
- 候補の根拠は、必要に応じて軽く参照できる形にする
  - 投稿者が「なぜ出た候補か」を判断できる程度でよい
  - 常時詳細説明を並べて、主入力を圧迫しない
- 候補 UI の第一軸は、将来的に `出所` ではなく `意味カテゴリ` とする
- 初期表示は厳選表示とし、候補棚を辞書ブラウザ化しない
- `行為名 / シチュ / 性癖` は、必要時のみ展開できる補助カテゴリへ寄せる

## 7. API 方針

Phase 1 では最小 API に限定する。

推奨:

- `GET /api/pixiv-tags/suggest`

入力例:

- `title`
- `caption_pixiv`
- `target_folder`
- `current_tags`

返却例:

```json
{
  "success": true,
  "message": "Pixiv tag suggestions loaded",
  "suggestions": [
    { "tag": "オリジナル", "source": "history", "score": 10 },
    { "tag": "ブルーアーカイブ", "source": "works", "score": 8 }
  ]
}
```

Phase 1 では `learn` API を必須にしない。

## 8. 学習仕様

### 8.1 Phase 1

学習は実装しない。

### 8.2 Phase 2

学習を導入する場合は、まず単純頻度のみとする。

学習トリガー候補:

- タスク編集モーダル保存時

### 8.3 学習の制約

- 学習は順位調整のみに使う
- 自動確定には使わない
- 未保存入力は学習しない

## 9. エラー時方針

- 辞書読込失敗時もタスク編集は継続可能とする
- 候補取得失敗時は Pixivタグ手入力へフォールバックする
- Pixivタグ支援の障害で Patreon / Discord / タスク一覧を巻き添え停止させない

## 10. 非目標

この仕様は以下を目的としない。

- Pixiv 全タグの完全収集
- 外部巨大タグ DB 依存
- AI による自由生成型タグ推論
- 自動投稿強化
- 他プラットフォーム向けタグ共通化
