# リメイク改訂実装計画

更新日: 2026-03-27

## 0. ゴール

- 既存機能と運用資産を守りながら、Web UI を中心に安全にリメイクする。
- 初回実装では「見た目の刷新」より先に、「正本定義」「非対話経路」「実行安全性」を揃える。
- 正本実装先は `E:\codex\workspace\projects\post-manager-remake\app` とする。
- workspace 外の旧バックアップは参照用バックアップとして扱う。

---

## 1. 実装方針

- 変更は `構造 -> 見た目 -> 安全性` の順で進める。
- CLI と Web の入口は温存する。
- `manager.py` を業務フローの中心に据えたまま、CLI 用対話経路と Web 用非対話経路を分ける。
- `data/posts.csv`、`config/*`、`browser_profile/` は運用資産として保護前提で扱う。
- 「処理完了」は「自動入力完了」であり、「投稿確定」ではない文脈を維持する。

## 3. フェーズ別計画

### Phase 1. 定義統一

目的:
Task / Settings / 実行状態の定義を単一化し、以降の実装の基準を揃える。

対象:
- `src/api_server.py`
- `src/manager.py`
- `src/utils.py`
- `web/script.js`
- `web/index.html`

やること:
- Task 項目と Settings 項目の単一ソースを決める
- schedule の変換責務を明文化する
- `all` を含む step 語彙の正を決める
- Web 用の表示状態とサーバー側実行状態の対応を決める

完了条件:
- 「どの値が永続」「どの値が派生」「どの値が UI 状態か」が表で説明できる
- API / UI / CSV の不一致が一覧化されている

### Phase 2. 実行経路の分離

目的:
CLI 対話経路と Web 非対話経路を分け、Web 実行で `input()` 依存を残さない。

対象:
- `src/manager.py`
- `src/preprocessor.py`
- 必要なら新規 service/helper モジュール

やること:
- skip/overwrite の判断を「CLI では対話」「Web では明示パラメータか安全デフォルト」に分ける
- shared browser context の扱いを service 化しやすい形へ寄せる
- 実行結果を UI に返しやすい戻り値へ寄せる

完了条件:
- Web から `clean/mega` を呼んでも標準入力待ちで止まらない設計になっている
- CLI の現行操作感を壊さない

### Phase 3. API / プロセス基盤の固定

目的:
Web UI が頼る API 契約と長時間処理の状態管理を安定化する。

対象:
- `src/api_server.py`
- `web/script.js`

やること:
- タスク CRUD の責務を整理
- プロセス状態の語彙を固定
- 実行中・完了・停止・失敗のレスポンス形を整理
- 進行表示の情報源を決める
- WebSocket ログを使うか、ポーリングで完結するかを確定

完了条件:
- UI が「何を見れば状態判断できるか」が一意になる
- `isProcessing` だけに依存しない設計方針が決まっている

### Phase 4. UI 構造リメイク

目的:
見た目より先に、情報構造と主操作導線を整理する。

対象:
- `web/index.html`
- `web/script.js`
- `web/style.css`

やること:
- タスク一覧、アクション、設定、ログの優先度を再配置
- 主操作を分かりやすくする
- タスク編集フォームの群分けを整理
- 実行中の案内、次の行動、危険操作の確認を組み込む

完了条件:
- 「どこを見れば何をすべきか」が初見で分かる
- 二重実行防止、エラー時の次アクション、ブラウザ確認案内が UI 上に存在する

### Phase 5. 見た目と状態安全性の仕上げ

目的:
構造を崩さずに視認性と誤操作防止を仕上げる。

対象:
- `web/style.css`
- `web/script.js`

やること:
- 階層、余白、視線誘導を調整
- 危険/進行中/完了/要確認の状態差を視覚化
- 文言を「次の行動が分かる」形に統一

完了条件:
- UI が構造先行で改善されている
- 状態差が色と文言の両方で伝わる

### Phase 6. 文書と切替

目的:
実装後に README と docs を現行の正へ揃え、段階切替できる状態にする。

対象:
- `README.md`
- `docs/*`

やること:
- `browser_profile/` などの現行正へ文書更新
- `all` や Discord 挙動の正を反映
- 切り戻し手順を残す

完了条件:
- 実装と文書のズレが埋まっている
- 切り戻し判断ができる

## 4. 変更単位の推奨分割

1. 定義統一だけの変更
2. 実行経路分離だけの変更
3. API/状態管理だけの変更
4. UI 構造変更
5. UI 見た目/安全性変更
6. 文書更新

### 分割理由

- 副作用の強い処理と UI の見た目変更を同一コミットに混ぜない
- 問題が起きたとき、どの層で壊れたか切り分けやすくする

---

## 5. 実装時の確認観点

### 正常系

- タスク追加/編集/削除が CSV に round-trip する
- 設定保存で空パスワード維持が効く
- `clean/mega/pixiv/patreon/discord` が既存の入口から動く
- Pixiv/Patreon が手動最終確認前提のまま残る

### 異常系

- CSV なし / 空タスク時の表示
- フォルダ不正時の案内
- 既存 zip / 既存 MEGA ファイル時の分岐
- browser profile ロック時の retry
- プロセス停止時の UI 表示

### 運用系

- `data/posts.csv` を壊さない
- `config/*` の保存で既存値を失わない
- `browser_profile/` を初期化しない
- Discord 重複送信の扱いを明示する
