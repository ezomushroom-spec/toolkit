## 3. 置き換え方針

### 3.1 推論構造

初回は、旧版と同様に「単一 `.pt` モデル + クラス別 ON/OFF」で構成する。

理由:

- 旧版の実績ある運用に最も近い
- まず性能・精度・整合性を保ちやすい
- `AutoMosaic` 側の ONNX 前提マルチモデル構造を無理に延命しなくてよい

初回では以下に寄せる。

- `processor.load_model(model_path)`
- `processor.has_model()`
- `processor.detect_candidates(img, imgsz, min_conf)`
- `processor.filter_candidates(raw_candidates, class_settings)`
- `processor.process_batch(...)`

### 3.2 UI 構造

`AutoMosaic` の PySide6 UI は活かす。
ただし、モデル指定 UI は「部位ごとの ONNX 選択」ではなく、まずは「単一 `.pt` モデル選択」に寄せる。

残す UI:

- 入出力フォルダ
- フィルタ種別
- 形状
- 推論サイズ
- 強度
- マージン
- 部位別 ON/OFF
- 部位別閾値
- プレビュー
- 一括処理

初回で削るもの:

- 部位ごとの個別モデル選択前提
- ONNX モデル説明

### 3.3 非同期方針

初回では最低限これを守る。

- 一括処理はワーカースレッド
- プレビュー推論は UI 固定化を避ける方向で設計する
- ローディング中は二重実行を防ぐ

ただし、最初の完成単位では「完全な高度非同期化」より、「旧版より悪化しないこと」を優先する。

## 5. Risks

- `AutoMosaic` は現在「部位別 ONNX モデル」前提のため、UI とコアの両方に修正が必要
- 旧版のクラス ID が数値、`AutoMosaic` のクラス設定が文字列 ID 前提のため、整合設計が必要
- プレビューの同期実行が残ると、UI フリーズ感が再発する可能性がある
- 手動削除した枠を一括処理へどう反映するかは、初回では未完でもよいが設計上は残課題
- 旧設定ファイルとの互換をどこまで持つかで実装量が変わる

## 6. Unknowns Requiring Final Judgment

- 初回は単一モデル前提に固定するか、将来のマルチモデル余地を残すか
- `settings.json` の旧 ONNX 形式を読む互換コードを入れるか
- 手動削除したプレビュー枠を初回から処理結果へ反映するか
- プレビュー推論を初回から完全非同期にするか、次段階に回すか

## 7. Suggested First Pass Scope

最小安全スコープは以下。

- `core.processor` を `.pt` ベースへ置換
- UI のモデル選択を単一 `.pt` モデル化
- 1 枚プレビューと一括処理を動かす
- 設定保存/復元と日本語パス対応を成立させる

これを超える改善は、第 2 パスに回す。

第 2 パス候補:

- プレビュー推論の本格非同期化
- 誤検知枠削除を処理結果へ反映
- 視認性と導線の改善
- 起動・配布導線の整理

## 8. 完了条件

第一パス完了の条件は次のとおり。

- `.pt` モデルを選んで起動後に再利用できる
- 対象部位の ON/OFF と閾値でプレビュー結果が変わる
- 一括処理が最後まで完走またはキャンセルできる
- 日本語パス画像を読み書きできる
- 旧版と比べて目的達成フローが壊れていない

## 9. 戻し方

- 旧版 `app_v18.py` は削除しない
- `AutoMosaic` の変更で問題が出ても、利用導線は旧版へ戻せるようにする
- ONNX 関連ファイルは初回では削除せず、参照停止に留める
