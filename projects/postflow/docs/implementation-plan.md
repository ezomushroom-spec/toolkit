# 実装計画

## 1. 依頼の要点

- 目的: 画像投稿の前工程専用のローカルデスクトップアプリを初版として成立させる
- 対象 project: `postflow`
- 変更したいこと: 作品管理、画像順管理、先頭画像指定、投稿先状態管理、投稿前確認を実装する
- 変えないこと: 元画像ファイルは編集しない。キャプションやタグ生成は扱わない

## 2. 現状確認

- 対象画面: 作品一覧、画像準備、投稿確認
- 関連処理: 作品 CRUD、画像登録、並び替え、先頭画像設定、状態更新、確認状態更新
- 関連データ / 設定: SQLite、ローカル画像パス、投稿先マスタ
- 正本: `projects/postflow/docs/`。実装開始後は `projects/postflow/app/`
- 現時点で分かっている制約: 完全ローカル、Windows 優先、少人数限定運用

## 3. 論点分解

- UI の論点: 一覧性、中央プレビュー優先、誤操作防止、空状態と異常系表示
- データ / 設定の論点: SQLite スキーマ、作品と画像の関係、全体ステータス再計算
- 外部 I/O の論点: ローカル画像読み込みのみ。投稿自体は外部アプリへ委譲
- 互換性の論点: 初版では新規 project のため過去互換はなし
- 運用上の論点: 元画像非破壊、危険操作確認、二重実行防止

## 4. 最小実装単位

### Step 1
- 目的: アプリの土台と DB 初期化を作る
- 触る場所: `app.py`, `main_window.py`, `config.py`, `db/`
- この段階で確認すること: アプリ起動、一覧画面表示、DB 作成
- 戻し方: DB 初期化前の雛形状態へ戻す

### Step 2
- 目的: Repository / Service 層を作り、UI なしで主要業務ルールを成立させる
- 触る場所: `repositories/`, `services/`
- この段階で確認すること: 作品作成、画像追加、並び替え、先頭画像設定、状態更新、確認状態更新
- 戻し方: Service 単位、Repository 単位で差し戻す

### Step 3
- 目的: 一覧画面を実装する
- 触る場所: `pages/works_list_page.py`, `widgets/work_card_widget.py`, `widgets/works_filter_bar.py`, `dialogs/create_work_dialog.py`, `dialogs/confirm_delete_work_dialog.py`
- この段階で確認すること: 作品作成、一覧表示、検索、絞り込み、削除
- 戻し方: 一覧画面だけを雛形へ戻す

### Step 4
- 目的: 画像準備画面を実装する
- 触る場所: `pages/work_editor_page.py`, `widgets/image_strip_widget.py`, `widgets/image_preview_widget.py`, `widgets/destination_status_panel.py`, `dialogs/confirm_remove_image_dialog.py`
- この段階で確認すること: 画像追加、除外、並び替え、先頭画像設定、投稿先状態更新
- 戻し方: 画像準備画面を一覧だけ運用へ戻す

### Step 5
- 目的: 投稿確認画面を実装する
- 触る場所: `pages/work_review_page.py`, `widgets/review_checklist_widget.py`
- この段階で確認すること: 先頭画像確認、画像順確認、確認チェック、投稿後状態更新
- 戻し方: 確認画面を外し、編集画面運用へ戻す

### Step 6
- 目的: 異常系、安全性、見た目調整を仕上げる
- 触る場所: 各画面、ダイアログ、`assets/styles/app.qss`
- この段階で確認すること: パス切れ、読込失敗、空状態、危険操作確認、二重実行防止、一貫した状態表示
- 戻し方: 機能は維持しつつ、スタイルや補助表示だけ段階的に戻す

## 5. 確認計画

- 正常系: 作品作成、画像追加、並び替え、先頭画像設定、投稿先状態更新、確認完了、再起動後保持
- 失敗系: 画像読込失敗、DB 保存失敗、存在しない `work_id`
- 空入力: 空の作品名
- 不正設定: 投稿先 0 件、有効投稿先なし
- 存在しないパス: パス切れ画像、削除済み画像
- 途中停止 / キャンセル: ダイアログキャンセル、画面戻り、確認途中での遷移
- 危険操作: 作品削除、画像除外、投稿済み状態の更新
- メモリ / GPU / 長時間処理: 大量画像のサムネイル表示時に過剰消費がないか

## 6. 未決事項

- 実装前に決めること: 実装フォルダ構成、SQLite ファイル配置、初期投稿先名称
- 実装しながら決めてよいこと: メモ保存タイミング、投稿日時の戻しルール、ドラッグ並び替えを初版に入れるか
- ユーザー確認が必要な分岐: 現時点ではなし

## 7. 完了条件

- この案件で完了と見なす条件:
  - 作品を作成できる
  - 画像を複数登録できる
  - 並び順を変更できる
  - 先頭画像を指定できる
  - 投稿先状態を更新できる
  - 確認画面で見直せる
  - 再起動後も保持される
  - パス切れや画像 0 件でも破綻しない
- 今回やらないこと:
  - キャプション / タグ / 説明文生成
  - 自動投稿
  - 予約投稿
  - 画像加工
  - クラウド同期
  - 配布最適化
- 残るリスクの扱い:
  - 実データ量が増えた場合の速度は、初版後に必要なら調整する
  - 配布や複数端末利用は次段階で再判断する
