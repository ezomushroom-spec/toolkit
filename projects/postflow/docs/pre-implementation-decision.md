# 実装前判断

## 1. 文書の目的

- 何を決める文書か: `PostFlow` 初版の実装方式、スコープ、正本、見送る案を固定する文書
- この文書を読む人: 実装担当者、引き継ぎ時の Codex、少人数の運用者
- この文書を読んだあとに迷わなくしたいこと: どの技術で、どこまで作り、何を初版で作らないか

## 2. 前提

- 対象 project: `postflow`
- 対象画面 / 機能: 作品一覧、画像準備、投稿確認
- 現在の正本候補: まだコード正本なし。文書正本は `projects/postflow/docs/`
- 実コード / 実データ / 現用設定で確認したこと: workspace に実装計画テンプレートと品質基準が存在する。`PostFlow` の既存コードはない
- 変えない条件: 完全ローカル運用、元画像非破壊、キャプション / タグ作成は対象外
- 再確認可能な条件: 将来の配布方法、エクスポート、細かな UI 表現

## 3. 候補比較

### 候補 A

- 概要: ブラウザ中心の Web アプリ
- 要件適合性: ローカル画像中心の用途にはやや遠回り
- 実装の確実性: 高いが、ローカルファイルとの相性で制約が増える
- 保守性 / 拡張性: 将来共有に広げやすい
- UI / 運用面の評価: 画面は作りやすいが、ローカル運用には余分な構成が乗りやすい
- 既存資産の再利用: 既存資産なし
- 主な弱点: 完全ローカル前提の初版には過剰

### 候補 B

- 概要: `Python + PySide6 + SQLite` のローカルデスクトップアプリ
- 要件適合性: 高い
- 実装の確実性: 高い
- 保守性 / 拡張性: 初版に十分。将来の拡張も許容可能
- UI / 運用面の評価: モダン寄りの業務ツール UI を構成可能で、ローカル画像との相性がよい
- 既存資産の再利用: Python 標準機能と軽量な構成を使える
- 主な弱点: Web 系より派手な演出はしにくい

### 候補 C

- 概要: ローカル Web + デスクトップ shell のハイブリッド
- 要件適合性: 高いが、初版にはやや複雑
- 実装の確実性: 中程度
- 保守性 / 拡張性: 将来の拡張には強い
- UI / 運用面の評価: 表現力は高いが、限定運用の内部ツールには構成が重い
- 既存資産の再利用: 既存資産なし
- 主な弱点: 初版で背負う複雑さに見合う利得が少ない

## 4. 採用判断

- 採用案: 候補 B の `Python + PySide6 + SQLite`
- 採用理由: 完全ローカル、少人数運用、ローカル画像中心という要件に最も素直で、初版 MVP を短い経路で実装できるため
- 見送る案と理由:
  - 候補 A は、ローカルファイル操作と運用の単純さで不利
  - 候補 C は、初版には構成が重い
- 読む先: `projects/postflow/docs/`
- 実装先: 実装開始後に `projects/postflow/app/`
- 文書更新先: `projects/postflow/docs/`

## 5. keep / fix / 後送り

### keep

- 維持する挙動: 元画像ファイル非破壊、一覧 → 準備 → 確認の導線、投稿先ごとの状態管理
- 維持する理由: 誤操作コストを下げ、利用目的に直結するため

### fix

- 今回直すもの: 投稿前の整理不足、順番確認のしにくさ、進行状態の分散
- 直す理由: 投稿漏れ、確認漏れ、管理の煩雑さを減らすため

### 今回やらないもの

- 後段へ送るもの: キャプション生成、自動投稿、画像加工、クラウド同期、配布最適化
- 後段へ送る理由: 初版の価値に直結せず、複雑さを増やすため

## 6. 実装前に守る運用ルール

- 保護対象データ / 設定: 元画像ファイル、将来の SQLite データ、投稿先状態
- 先に確認する危険操作: 作品削除、画像除外、投稿済み状態の上書き
- 触る範囲: `projects/postflow/` 配下の文書と、実装開始後の project コード
- 触らない範囲: `archive/`, `build/`, `dist/`, `*_profile`, `src_backup_*`
- 戻し方: フェーズ単位で巻き戻せる構成にし、元画像には触れない
