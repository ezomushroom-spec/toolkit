# Project Rules

## 1. この project の目的

- `PostFlow` は、画像投稿の前工程を整理するローカルデスクトップアプリです。
- 主な利用者は、ローカル環境で少人数運用を行う利用者です。
- 完了時に守るべき価値は、投稿漏れと確認漏れの低減、元画像の保全、誤操作防止です。

## 2. 正本

- 実装の正本: `projects/postflow/app/`
- 設定の正本: `projects/postflow/app/postflow/config.py`
- データの正本: `projects/postflow/runtime/postflow.db`
- 編集してはいけない生成物やバックアップ: `archive/`, `build/`, `dist/`, `*_profile`, `src_backup_*`

## 3. 壊してはいけない既存挙動

- 主操作: 作品作成、画像追加、並び替え、先頭画像設定、投稿先状態更新、投稿前確認
- よく使う導線: 一覧 → 画像準備 → 投稿確認
- 互換維持が必要な設定や保存形式: 初版では SQLite とローカル画像パス参照
- 外してはいけない運用制約: 元画像ファイルを削除しない

## 4. build / run / test / lint

- 開発起動: `python app/app.py`
- Windows 起動バッチ: `start.bat`
- Windows デバッグ起動バッチ: `debug-start.bat`
- 本番起動: 未定
- テスト: 必要に応じて `python -m py_compile ...` と手動確認
- lint / format: 未定
- 実行に必要な前提: Python 3.11+, PySide6, SQLite

## 5. 優先確認項目

- 正常系: 作品作成、画像追加、並び替え、先頭画像設定、投稿先状態更新、再起動後保持
- 失敗系: 画像読込失敗、DB 保存失敗
- 空入力: 空の作品名
- 不正設定: 投稿先未設定
- 存在しないパス: パス切れ画像
- 途中停止やキャンセル: ダイアログキャンセル、画面遷移戻り
- 危険操作: 作品削除、画像除外、投稿済み状態の上書き
- メモリ / GPU / 長時間処理: 大量画像読込時のメモリ使用

## 6. UI / 操作上の注意

- 主操作の位置や表現: 各画面で主操作は 1 つだけ強調する
- 危険操作の扱い: 対象明示付き確認ダイアログを必須にする
- エラー表示の要件: 次の行動が分かる日本語メッセージにする
- 二重実行防止: 保存、状態更新、追加処理で二重実行を防ぐ

## 7. 実装時の注意

- 無関係なリファクタを混ぜない範囲: 初版 MVP に必要な範囲だけ触る
- 触る前に確認すべき関連処理: Service 層の状態再計算、Repository 層の整合性
- 段階移行が必要な箇所: UI は一覧 → 準備 → 確認の順で段階実装
- 戻し方: フェーズごとに戻し、元画像ファイルは常に非破壊で保つ
- 実装前の方式整理、仕様更新、計画更新、project 文書作成はユーザーへの逐一確認なしで進めてよい
- ユーザー確認が必要なのは、元画像を壊す可能性がある操作、初版範囲を超える方針変更、要件の意味が変わる分岐に限る

## 8. 変更時の必須セット作業

- 更新する docs: `docs/current-state-check.md`, `docs/implementation-plan.md`, 必要なら `START_HERE.md` と `PROJECT_SUMMARY.md`
- 最低限の確認コマンド: `python -m py_compile app\\app.py app\\postflow\\main_window.py`
- 最低限の手動確認: 一覧から作品を作り、画像を追加し、状態更新まで一周する
- 失敗時に先に見るログやファイル: SQLite ファイル、画像パス、アプリ例外ログ
- Windows 向け project では原則 `start.bat` と `debug-start.bat` を同梱し、起動確認を行う

## 9. 完了報告で必ず触れること

- 何を変えたか
- 何を確認したか
- 何が未確認か
- 残るリスク
