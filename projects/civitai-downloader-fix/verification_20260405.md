# 確認結果 2026-04-05

## 何を変えたか

- `download_jobs` を起動時に毎回削除しないように修正した
- バージョン解決失敗時にジョブを `failed` にし、エラー文言を保持するようにした
- 未解決ジョブの `retry` で再解決へ戻せるようにした
- ダウンロード一覧の行マッピングを表示順基準に修正した
- ジョブ削除に確認ダイアログを追加した
- API 接続テストを UI スレッド外で実行するようにした
- 削除済みジョブの完了通知後もキュー処理が止まらないようにした
- Qt WebEngine の破損キャッシュ/共有辞書を段階退避し、新しい保存先へ切り替えるようにした
- 終了時に更新チェックとダウンロード系シグナルが DB クローズ後へ流れにくいよう停止待ちを追加した
- キャンセル/削除時に `temp_path` と `.part` を掃除するようにした
- 通知履歴を一括削除できるようにし、最新100件だけ自動保持するようにした
- 一括追加ダイアログで同一モデルの重複表示を除外し、選択件数を即時追従させた
- 通知タブの手動チェックに実行中表示、二重実行防止、前回確認時刻、検出件数を追加した
- ダウンロードタブに状態要約と簡易フィルタを追加した
- 失敗ジョブから失敗詳細とログフォルダを開けるようにした
- WebEngine の HTTP キャッシュを起動ごとの runtime パスへ分離し、強制終了後のロック済みキャッシュを踏みにくくした

## 確認結果

- 正常系
  - 実 GUI プロセスは起動後 5 秒時点で生存していた
  - DB 再初期化で既存ジョブは保持された
  - 未解決失敗ジョブの再試行導線は機械確認で成立した
  - 行マッピングは削除後も表示順と一致した
  - Qt WebEngine の起動時エラーは、キャッシュ/共有辞書退避後の再起動で再現しなかった
  - shutdown ガード追加後も実 GUI プロセスは起動後 5 秒時点で生存し、stderr は空だった
  - キャンセル/削除時の temp 掃除は機械確認で成立した
  - 利便性改善 Phase 3 の部品レベル確認と短時間起動確認は成立した
- 失敗系
  - バージョン解決失敗時に失敗状態へ落とすコード経路を追加した
  - API 接続テストの通信失敗は UI スレッド外で扱うようにした
- UI 安全性
  - ジョブ削除前に確認を入れた
  - `No` 選択時に削除されないことを GUI 部品レベルで確認した
  - API 接続テスト中に操作ボタンが即時無効化されることを確認した
- 操作性
  - 保存先の実体をダウンロードタブ上からツールチップで確認できるようにした
  - 設定タブのパス指定をダブルクリック依存から `参照` ボタンでも実行できるようにした
  - 通知履歴を手動で全削除でき、履歴が増えすぎないよう最新100件に制限した
  - 一括追加、通知チェック、ダウンロード一覧、失敗復旧の主要導線を前面化した
  - 範囲選択またはチェック済み複数行から右クリック削除した場合、右クリック行だけでなく選択範囲全体を削除対象にした
  - 設定タブで追加・編集したプリセットが、保存ボタン待ちなしでダウンロードタブへ反映されるようにした

## 残るリスク

- 旧 `download_jobs` スキーマが大きく異なる DB に対しては、今回の軽量 migration だけでは不足する可能性がある
- GUI 実操作の未確認部分では、文言やタイミング差による小さな副作用が残る可能性がある
- 長時間ダウンロードや大量ジョブでの挙動は今回未計測
- 既存の WebEngine キャッシュを退避するため、初回起動時だけブラウザ系の一時データ再生成が走る
- `QThreadPool.globalInstance()` を停止待ちするため、終了時に進行中タスクぶんだけ待機が入る
- WebEngine の runtime cache は起動ごとに再生成されるため、HTTP キャッシュによる高速化は期待しない
