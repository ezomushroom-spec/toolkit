# 実装計画

## 1. 依頼の要点

- 目的: `E:/civitai_downloader` を workspace 管理の案件として巻き取り、危険度の高い不具合を修正する
- 対象 project: `civitai-downloader-fix`
- 変更したいこと: データ消失、詰み状態、誤操作、UI フリーズの抑制
- 変えないこと: アプリ全体の構成、主要タブ構成、既存保存形式

## 2. 現状確認

- 対象画面: ブラウズ、ダウンロード、通知、設定
- 関連処理: DB 初期化、バージョン解決、キュー一覧更新、API 接続テスト、削除操作
- 関連データ / 設定: `download_jobs`, `settings`, `folder_presets`, `tracked_models`, `notifications`
- 正本: `E:/civitai_downloader`
- 現時点で分かっている制約: 実運用データを壊さない、既存導線を残す、外部 API と GUI を含むため全自動検証は限定的

## 3. 論点分解

- UI の論点: 削除確認、失敗理由表示、フリーズ回避
- データ / 設定の論点: 起動時のテーブル削除防止、互換維持
- 外部 I/O の論点: API 失敗時の扱い、ファイル移動失敗時の退避
- 互換性の論点: 既存 DB と設定値を維持したまま修正できるか
- 運用上の論点: 再起動後の保持、詰んだジョブの復旧しやすさ

## 4. 最小実装単位

### Step 1
- 目的: DB 初期化の破壊挙動を止める
- 触る場所: `E:/civitai_downloader/db/models.py`
- この段階で確認すること: 既存 `download_jobs` を消さずに初期化が通ること
- 戻し方: `db/models.py` の変更を差分単位で戻す

### Step 2
- 目的: バージョン解決失敗時の詰み状態を解消する
- 触る場所: `E:/civitai_downloader/core/resolve_worker.py`, `E:/civitai_downloader/core/download_manager.py`, `E:/civitai_downloader/ui/download_tab.py`
- この段階で確認すること: 失敗理由が `download_jobs.error_message` と UI に反映されること
- 戻し方: 解決失敗時の状態更新ロジックを差分単位で戻す

### Step 3
- 目的: 一覧更新と危険操作の安全性を上げる
- 触る場所: `E:/civitai_downloader/ui/download_tab.py`, `E:/civitai_downloader/ui/settings_tab.py`
- この段階で確認すること: 行マッピングが表示順と一致すること、削除確認が出ること、API テストが UI を塞ぎにくくなること
- 戻し方: UI 局所差分をファイル単位で戻す

## 5. 確認計画

- 正常系: 初期化、ジョブ追加、ジョブ開始、設定保存
- 失敗系: バージョン解決失敗、API 接続失敗、保存先移動失敗
- 空入力: API キー未入力
- 不正設定: 存在しないプリセットパス
- 存在しないパス: 保存先ディレクトリ不在
- 途中停止 / キャンセル: キャンセル後状態、削除前確認
- 危険操作: ジョブ削除
- メモリ / GPU / 長時間処理: 長時間 GUI 起動は今回は限定確認

## 6. 未決事項

- 実装前に決めること: 第1段階の修正対象は上記 3 点に絞る
- 実装しながら決めてよいこと: 軽量 migration の必要有無、UI 文言の詳細
- ユーザー確認が必要な分岐: データ形式変更や全面 UI 変更が必要になった場合

## 7. 完了条件

- この案件で完了と見なす条件: DB 初期化破壊が止まり、解決失敗ジョブが詰まず、一覧誤操作リスクが下がり、確認結果を文書化する
