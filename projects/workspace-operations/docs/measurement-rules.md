# 計測ルール

この文書は、workspace の実効性計測で何を見るか、どこから拾うか、どの頻度でまとめるかを固定するためのルールである。

## 1. 対象

最初に見る対象は次の 4 本に限定する。

1. skill 利用の偏り
2. 完了報告の充足度
3. docs 更新漏れ
4. 詰まりの型

## 2. 拾う元

### skill 利用の偏り

- 主な取得元: 会話ログ
- 補助取得元: 完了報告、案件 docs

残し方:

- 1 案件または 1 大きな依頼ごとに、使った skill 名を列挙する
- 記録先は原則として `skill-usage-log-YYYYMM.md` にする

### 完了報告の充足度

- 主な取得元: 完了報告

見る項目:

- 何を変えたか
- 何を確認したか
- 何が未確認か
- 残るリスク

残し方:

- 各項目が埋まっているかを簡易に確認する

### docs 更新漏れ

- 主な取得元: 完了報告、案件 docs

見る項目:

- docs 更新要否を見たか
- 中規模以上で判断メモや計画が残っているか

### 詰まりの型

- 主な取得元: 会話ログ、案件 docs、レビュー文書

分類:

- 正本未確定
- 仕様不足
- 設計分離不足
- 確認項目不足
- 実装後の副作用
- UI / 操作性不足
- docs 更新漏れ

## 3. 集計頻度

最初の集計頻度は次のどちらかにする。

- 月 1 回
- または大きな改善フェーズの区切りごと

まずは月次を基本にする。

## 4. 記録物

この project で使う記録物は次の 3 つ。

- `measurement-rules.md`
- `measurement-snapshot-YYYYMM.md`
- `skill-usage-log-YYYYMM.md`

## 5. 今は見ないもの

最初は次を対象外にする。

- 厳密なルール違反率
- すべての会話の全件分析
- skill ごとの精密な成功率
- 工数や時間の厳密集計
- 自動イベント収集

## 6. スナップショットの最小項目

毎回のスナップショットには少なくとも次を入れる。

- 期間
- 主な案件
- 使われた skill
- 完了報告の抜けやすい項目
- docs 更新漏れの有無
- よく出た詰まり
- 次に直すべき運用上の弱点

## 7. 運用上の注意

- 数字だけで評価しない
- 件数が少ない月は、傾向が弱いことを明記する
- 明らかな改善候補があっても、1 回の観測だけで大きく rule を変えない
- 自動化は、手動スナップショットが 2〜3 回続いた後に検討する
- skill 利用は厳密な頻度集計より、偏りや未使用を見つけるために使う
