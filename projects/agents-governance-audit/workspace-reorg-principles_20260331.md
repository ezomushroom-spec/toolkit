# Workspace Reorg Principles

## 現状確認

- workspace の大枠分離は成立している
- 問題は「完全に壊れている構造」よりも「入口不統一」と「境界未定義」に寄っている
- したがって、再編は大移動よりも先に基準整備から入るべき

## 採用判断

### 採用する進め方

1. 入口文書を整備する
2. active / backup / archive の基準を明文化する
3. `docs` の正式文書とテンプレートの責務を分ける
4. 旧構成→新構成の対応表を作ってから必要最小限の移動を行う
5. 物理移動は、参照更新と戻し方が整理できたものだけに限定する

### 今すぐ全面再編しない理由

- 現在の root / docs / projects / .agents / .codex の大枠は破綻していない
- 先に移動すると、active と backup の判断が曖昧なまま物理変更が先行する
- 入口文書がない状態で場所だけ変えると、初見の読み順がむしろ悪化する
- backup や README を含む既存参照に対し、対応表なしで変更すると追跡性が落ちる

### 第1優先で整える対象

- 各 project の `START_HERE.md` または同等の入口文書
- backup と active の判定基準
- `archive` に送る対象の判断基準
- `docs` のうち正式運用文書とテンプレートの分類

### 第2優先で整える対象

- `docs` のサブフォルダ化
- backup 群の段階的な退避
- project ごとの brief 文書命名統一

### 第3優先で整える対象

- 大規模なフォルダ再配置
- 名前変更を伴う整理
- 全 project 一括の文書名統一

## 最小実装単位

### Step 1

- 目的: 再編判断の基準を固定する
- 対象: `projects/agents-governance-audit` の調査文書
- 確認点: 現状確認、採用判断、未決事項が分かれていること
- 戻し方: 文書を参照しないだけで戻せる

### Step 2

- 目的: project 入口文書の標準形を決める
- 対象: `docs/templates/project-AGENTS.template.md` と新規入口テンプレート候補
- 確認点: project ごとの最初の読み順が明確になること
- 戻し方: テンプレート運用を採用しない

### Step 3

- 目的: active / backup / archive の対応表を作る
- 対象: `projects` と `archive`
- 確認点: 物理移動前に旧→新の追跡表があること
- 戻し方: 対応表だけ残して移動は保留

### Step 4

- 目的: 小さく安全な物理再編を行う
- 対象: 基準が固まった文書や backup フォルダ
- 確認点: 参照更新、理由、戻し方が残ること
- 戻し方: 対応表を使って元位置へ戻す

## 未決事項

- `README.md` を入口として許容する基準を作るか
- backup フォルダを `archive/projects-backups` のような置き場へ分離するか
- `docs/templates` の新設を今やるか、文書数が増えてからやるか
