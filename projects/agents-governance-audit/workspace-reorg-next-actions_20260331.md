## 採用判断

次に着手する候補を優先順で並べる。

1. `START_HERE.md` の雛形を作る
2. `projects` の各主要 project に入口文書を段階投入する
3. backup / archive 判定ルールを文書化する
4. その後にだけ、対応表を作って物理移動を検討する

### 理由

- 初見の読み順が整うと、再編前でも運用事故が減る
- backup と active の境界が見えると、物理移動の判断がしやすくなる
- 参照更新が必要な変更を後ろへ回せるため、戻しやすい
