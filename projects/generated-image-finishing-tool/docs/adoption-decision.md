# 採用判断

## 採用する方向

- 実行エンジン層は Python を正本にする
- 初期版の処理実装は CPU 正本で成立させる
- 重い工程だけ将来的に GPU を段階導入できる構造にする
- UI は機能重視のデスクトップ GUI を採る
- レシピ形式は JSON を採る

## 採用理由

### 1. 専門性を最優先できる

- 画像仕上げ工程そのものを Python 側で詰めやすい
- 工程の追加、順序制御、パラメータ検証を素直に組める
- レシピ再現性と一括適用の整合を保ちやすい

### 2. 機能的な GUI を無理なく実現しやすい

- 工程一覧、比較表示、保存導線、一括適用、進捗表示をデスクトップ向けに整理しやすい
- ローカルファイル操作、日本語パス、保存先選択などを扱いやすい

### 3. 安定した動作を優先しやすい

- CPU 正本により GPU 非対応環境でも成立する
- GPU 失敗時の CPU フォールバックを設計しやすい
- 単画像適用と一括適用で同じ処理コアを使いやすい

## 採用構成

### 実行エンジン層

- Python ベース
- 想定ライブラリ: NumPy / OpenCV / Pillow
- 構造:
  - `Recipe`
  - `RecipeStep`
  - `ParameterSchema`
  - `RecipeExecutor`
  - `BaseStepExecutor`
  - 各工程実装

### UI 層

- 第一候補: PySide6 系デスクトップ GUI
- 役割:
  - 画像表示
  - 工程編集
  - 比較表示
  - レシピ保存 / 読込
  - 一括適用
  - 進捗表示
  - エラー表示

## レシピモデルの採用判断

- レシピは順序付き工程列とする
- 実行順は `steps` 配列順を正本にする
- `order_index` は初期版では持たない
- 1 レシピ = 1 JSON ファイル
- version を持たせ、将来のフォーマット変更に備える

## 一括適用の採用判断

- 一括適用は初期版の必須機能に含める
- 1 枚失敗で全停止しない
- 失敗対象は個別記録し、残りは継続する
- 中断要求を受け付ける
- 出力は入力元と分離し、元画像は破壊しない

## UI 構成の採用判断

- 主画面はレシピ編集画面とする
- 一括適用は主画面から分離した専用画面にする
- 設定はレシピと分離した専用画面にする
- GUI は見た目より、比較、工程編集、保存、一括実行のしやすさを優先する

## 見送った案と理由

### Web UI + Electron / Tauri

- 比較表示や見た目の自由度は高い
- ただし初期版では、処理コアとの橋渡し、ローカルファイル I/O、プレビュー受け渡しの複雑さが増える
- 今回の優先順位では第一候補にしない

### 最初から GPU 中心

- 高速化余地は大きい
- ただし環境差分、依存管理、フォールバック設計が重く、初期版の安定性と実装確実性を下げやすい
- CPU 正本を先に成立させる方が妥当

### GEGL 型の全面採用

- 工程モデルの思想としては有力
- ただし初期版に必要なのは自由なノードグラフではなく、順序付き工程列の確実な運用
- 現時点では主設計から外し、必要性が出た時に再評価する
