## 1. keep / fix 確定案

| 項目 | 判定 | 理由 | 実装着手時の扱い |
| --- | --- | --- | --- |
| CLI 入口の維持 | keep | 既存運用入口であり、Web と同時に崩すと切り戻しが難しい | 初回リメイクでは触らない |
| Web 入口の維持 | keep | UI は変えても起動方法は維持したい | 初回リメイクでは `start_webui.bat` を維持 |
| CSV 列名と列順 | keep | round-trip 破壊のリスクが高い | 列追加/削除/改名を先行しない |
| Web の index ベース更新 | keep | 改善余地は大きいが、初回で ID 移行まで入れると scope 超過 | 内部設計だけ用意し、初回は互換維持 |
| Task 項目群 | keep | 既存 UI/API/CSV を一気に崩さないため | 単一定義表に統合してから実装 |
| Settings 項目群 | keep | secrets/templates の現行運用を保持する | 読込経路は整理しても、保存形式は維持 |
| `mega_password` 空欄維持ルール | keep | 安全性と運用上の期待値が高い | UI 変更後も必ず残す |
| Pixiv/Patreon の手動最終確認 | keep | プロジェクトの安全要件 | 自動投稿確定は行わない |
| shared browser context | keep | profile 運用と確認フローの中核 | 初回では壊さない |
| profile ロック時の retry 導線 | keep | 実運用で重要 | 文言改善は可、導線削除は不可 |
| Web の `runAllSteps = clean/mega/discord` | keep | 前処理まとめ実行として扱う | UI 文脈を維持する |
| Discord で `zip_url` を送らない挙動 | fix | README と期待値に反する可能性が高い | 既知差分として明示してから修正 |
| Clean/MEGA の `input()` 対話 | fix | Web 経路では不整合でハング要因 | CLI 用と Web 用に分離する |
| WebSocket ログ未接続 | fix 候補 | 骨格だけあり、中途半端 | 初回はポーリング維持でも可。後で選択 |
| `count,image_count` の扱い不明瞭 | fix 候補 | sample と実装の責務が揺れている | 初回で役割を明文化し、保存可否を決める |
| README の `user_data/` / Enter待ち説明 | doc fix | 実装とずれている | 実装と並行で更新する |
| 古い設計書の単一 `caption` 前提 | doc fix | 既に媒体別へ変化済み | 実装と並行で更新する |

### keep/fix の前提ルール

- `keep`: 初回リメイク中に振る舞いを変えない
- `fix`: 初回リメイクの対象に含めるが、仕様確定を先に行う
- `fix 候補`: 初回に入れるかは工数を見て判断するが、少なくとも見える化する
- `doc fix`: 実装の正に合わせて文書更新する
- `要判断`: 不具合扱いに進まず、現UI文脈と安全要件を含めて先に正を決める
