# モザイクかけ太郎 再構築準備メモ

## 1. 目的

`モザイクかけ太郎` を全面的に再構築するにあたり、実装前の前提を固定する。
この文書の目的は、次の実装で判断がぶれないように、正本、対象、実装順、確認点、戻し方を明示すること。

今回の再構築では、UI の分離と保守性向上を進める一方で、推論基盤は `ONNX` を採用しない。
運用時パフォーマンスの観点から、推論の正本は `.pt + ultralytics / PyTorch` 系を維持する。

## 2. 前提の確定事項

- 現行運用正本は `app_v18.py`
- 次期正本候補は `AutoMosaic/`
- `AutoMosaic_YOLO_v1.0/` は比較用または試行系として扱い、次期正本にはしない
- 既存機能は削除前提で扱わず、段階移行する
- 実装順は `調査 -> 論点分解 -> 最小実装単位の確定 -> 実装 -> 確認` を基本とする
- UI 改善は `構造 -> 見た目 -> 安全性` の順で進める
- バックエンド相当の業務ロジックは shell 側や UI 側へ複製せず、コア処理に集約する

## 3. 今回の重要判断

### 3.1 ONNX は採用しない

理由:

- 運用時のパフォーマンス要件に合わない
- 現行実績のある `.pt` 運用から外れる
- 変換、検証、配布の管理点が増える

このため、以下は再構築の正本から外す。

- `onnxruntime` 前提の推論実装
- ONNX モデル変換フロー
- ONNX 専用テストや補助バッチ

### 3.2 再構築の主眼

再構築の主眼は、見た目の刷新そのものではなく、次の運用フローを改善すること。

`画像を選ぶ -> AI で検出する -> 人がプレビューで確認/必要なら修正する -> 一括出力する`

この流れを、旧版より安全に、速く、保守しやすくする。

役割:

### 4.3 比較用/非正本

役割:

- 今後の正本にはしない

## 5. 再構築で維持する必須機能

初期リリース段階で、少なくとも以下は落とさない。

- 入力フォルダ選択
- 出力フォルダ選択
- `.pt` モデル読込
- 対象部位ごとの ON/OFF
- 検知感度調整
- モザイク/ぼかし/黒塗り/白塗り
- Box と Mask の適用切替
- プレビュー更新
- 一括処理
- 処理中キャンセル
- 設定保存と復元
- 日本語パス対応
- Windows 上での起動導線

後回しにしてよい候補:

- UI の見た目の大幅演出
- 配布形式の最適化
- 旧試行系ディレクトリの整理
- 高度な手動追加 UI

優先実装候補:

- 誤検知結果の個別削除
- 非同期プレビュー
- 二重実行防止
- 分かりやすいエラー表示

## 6. 新アーキテクチャ方針

### 6.1 `core`

責務:

- `.pt` モデルの読込と推論
- 推論パラメータ管理
- 候補領域の統合と閾値フィルタ
- 一括処理の実行
- UI 非依存の例外整理

避けること:

- Widget 参照を持たない
- 画面状態に依存しない

### 6.2 `utils`

責務:

- 日本語パス対応の画像 I/O
- パス解決
- 汎用設定保存

避けること:

- 推論ロジックを持たない
- UI ロジックを持たない

### 6.3 `ui`

責務:

- 画面構成
- 状態入力
- プレビュー表示
- 実行中表示
- 危険操作確認
- エラー表示

避けること:

- 推論処理の本体を持たない
- ファイル保存や検出処理を直接抱え込まない

## 7. 実装順

## フェーズ 0: 準備固定

対象:

- 正本の定義
- 維持機能一覧
- ONNX 除外方針

確認点:

- 読むべき正本が 1 本化されている
- 何を残すかが明文化されている

戻し方:

- 本文書を基準に見直し

## フェーズ 1: コア処理の正本化

対象:

- 旧版の `.pt` 推論処理
- 画像加工処理
- 日本語パス I/O
- 設定保存

実装順の理由:

- UI を先に触ると、推論差分の問題が隠れるため
- 出力の正しさを先に固めるため

確認点:

- 旧版と同条件で推論結果が大きく崩れない
- 一括処理で出力画像が生成できる
- 日本語パスで読み書きできる

戻し方:

- 旧版の `app_v18.py` を比較基準として残す

## フェーズ 2: 最小 UI の再構築

対象:

- フォルダ設定
- モデル設定
- パラメータ設定
- プレビュー表示
- 一括処理起動

実装順の理由:

- 最小操作で動く骨格を先に作るため
- 見た目の作り込みより、操作導線を先に固めるため

確認点:

- 主操作が迷わず辿れる
- 処理開始とキャンセルが分かりやすい
- ローディング中に二重実行されない

戻し方:

- UI だけ差し替えてもコア処理は温存できる構造にする

## フェーズ 3: 安全性とレビュー導線

対象:

- 非同期プレビュー
- 誤検知枠の個別削除
- エラー表示
- 進行状況表示

実装順の理由:

- 体感品質に直結するため
- 「AI の自動化を人が補正できる」ことが価値の中心だから

確認点:

- UI が固まりにくい
- 誤検知をその場で消せる
- エラー時に次の行動が分かる

戻し方:

- 手動修正導線を無効化しても最低限のプレビュー運用が成立するようにする

## フェーズ 4: 配布と安定化

対象:

- 起動バッチ
- ビルド導線
- ログ整理
- リリース前確認

実装順の理由:

- 中身が固まる前に配布を詰めても手戻りが多いため

確認点:

- Windows 上で起動できる
- 必要な依存不足時に原因が分かる
- 既存利用者が迷わない

戻し方:

- 旧版 launcher と並行運用できる状態を残す

## 8. 最小実装単位

一度に広げすぎないため、最初の完了単位を以下に置く。

1. `.pt` モデルを読み込める
2. 1 枚の画像に対して検出してプレビュー表示できる
3. 同じ設定で一括出力できる
4. 日本語パスとエラー表示が成立する
5. 設定を保存して次回復元できる

この 5 点が揃うまでは、追加機能を広げすぎない。

## 9. 検証観点

実装後レビューでは少なくとも次を確認する。

- 旧版との挙動差分
- 危険操作の確認表示
- 二重実行防止
- エラー表示の分かりやすさ
- 未考慮ケース
- 日本語パス
- 空フォルダ
- モデル未読込時の案内
- 画像破損時の扱い
- キャンセル時の復帰

## 10. 今の時点でやらないこと

- ONNX ベース再設計
- ONNX 変換バッチの正本化
- 旧版機能の削除
- 配布方式の一本化
- 試行系フォルダの即削除

## 11. 次の実装着手点

次に着手するなら、最初の実装対象は以下が妥当。

- `AutoMosaic/` を次期正本として扱う前提で、ONNX 依存を外した `.pt` ベースの `core` 再定義
- 旧版から流用できる画像加工と I/O の切り出し
- 最小 UI で 1 枚プレビューと一括処理が動く状態づくり

## 12. メモ

`AutoMosaic/` は現時点では ONNX 前提の試作要素を含むため、そのまま完成候補として進めない。
使うのは主に UI 分割の方向性とディレクトリ構成であり、推論基盤は旧版の実績ある `.pt` 運用へ戻して組み直す。
