# 運用前提と保守ガイド

## 1. この案件の正本と非正本

### 正本

- `src/`
- `web/`

### 非正本

- `src_backup_*`
- `*_profile/` のバックアップや派生プロファイル
- `__pycache__/`

この案件では、バックアップやブラウザプロファイルが同居しているため、編集対象を最初に特定する。

## 2. 起動入口と責務

### CLI

- `start_cli.bat`
- `src/manager.py`

### Web

- `start_webui.bat`
- `src/api_server.py`
- `web/index.html`
- `web/script.js`
- `web/style.css`

### 原則

- 業務フローの本体は `manager.py`
- Web 側は API と UI を担当する
- 業務ロジックを CLI と Web に重複配置しない

## 3. データと設定の所有者

### `data/posts.csv`

- 投稿タスクの正本
- 人が編集する列と処理が埋める列が混在する

### `config/secrets.yaml`

- 認証情報の正本
- Git 追跡しない

### `config/templates.yaml`

- 各媒体向けテンプレートの正本

### `browser_profile/`

- 実運用のログイン状態を持つプロフィール
- 誤削除や初期化の影響が大きい

## 4. CSV 列の扱い

### 人が主に編集する列

- `target_folder`
- `title`
- `caption_pixiv`
- `caption_patreon`
- `caption_discord`
- `tags`
- `schedule`
- `zip_password`
- `patreon_tier`
- `discord_channel`

### 処理が埋める列

- `zip_url`

### 実行時派生になりやすい列

- `count`
- `image_count`

列を増減する場合は、CSV、API モデル、Web フォーム、サンプル CSV を揃える。

## 5. 処理フローと副作用

### Clean

- Pixiv 用画像を上書きでクリーン化する

### Zip

- Patreon 用画像をクリーン化し、`dist/` に Zip を作る

### MEGA Upload

- 外部ストレージにアップロードする
- `zip_url` を CSV に反映する

### Pixiv / Patreon

- ブラウザ自動入力まで
- 最終投稿確定は人が行う

### Discord

- Webhook 送信を行う
- 再送時は重複通知に注意する

## 6. ブラウザプロフィール運用

- 現行コードは `browser_profile/` を主に使用する
- README や古い設計書には `user_data/` 記述が残る場合がある
- 保守時は「現在どのプロフィールが正本か」を先に確認する
- Chrome の多重起動やプロフィールロック時は再試行導線を優先する

## 7. 保守時の確認ポイント

- 変更対象は CLI / Web / 両方のどれか
- `manager.py` との整合は取れているか
- CSV の round-trip は壊れていないか
- 人手確認フローは残っているか
- 外部状態変更の戻し方を説明できるか
