## 2. 既知差分・既知不具合表

| 論点 | 現行コードの実際 | README / docs の記述 | 暫定判定 |
| --- | --- | --- | --- |
| Web の全ステップ実行 | `clean -> mega -> discord` のみ | README の `--all` は全ステップ実行 | 要判断 |
| CLI の全ステップ実行 | `clean, mega, pixiv, patreon, discord` 全部有効化 | README と一致 | keep |
| ブラウザ profile 名 | `browser_profile/` を使用 | README には `user_data/` 記載あり | doc fix |
| キャプション設計 | 媒体別 `caption_pixiv / patreon / discord` | 古い設計書は単一 `caption` 前提 | doc fix |
| Discord への zip URL | 現行は `send_discord_notification(..., zip_url=None)` | README はダウンロードリンク利用前提 | fix 候補 |
| WebSocket ログ | API 側に受け口あり | UI では未接続 | keep/fix 要判定 |
| Clean / MEGA の対話 | CLI `input()` で skip/overwrite を確認 | Web 用非対話仕様は未整理 | fix 必須 |
| タスク識別 | Web は index ベース | 永続 ID の明記なし | 改善候補 |
| `count,image_count` | sample CSV には列あり | docs では主列扱いが揺れる | 要整理 |
| Enter 待ちの説明 | README では Enter で次へ進む説明 | 現行は Pixiv/Patreon 画面を準備後、ブラウザ確認を促す構成に変化 | doc fix |

### 暫定ルール

- `keep`: まず現行挙動を保持対象にする
- `fix 候補`: リメイク時に直したいが、仕様として明文化してから触る
- `doc fix`: 実装より先に文書側の正を更新する
- `要判断`: 未実装や不具合と断定せず、現UI文脈を含めて先に正を決める

## 3. 副作用 / 再実行 / 戻し方マトリクス

| 処理 | 書き換える対象 | 外部副作用 | 再実行時の主リスク | 最低限の戻し方 |
| --- | --- | --- | --- | --- |
| タスク追加/更新/削除 | `data/posts.csv` | なし | 列崩れ、誤削除 | 事前バックアップまたは差分復元 |
| 設定保存 | `config/secrets.yaml`, `config/templates.yaml` | 認証情報/テンプレートの上書き | 空欄保存、設定喪失 | 保存前退避、空欄維持ルール厳守 |
| Clean | Pixiv 元画像 | 画像の上書き | 原本喪失、再圧縮劣化 | 元画像バックアップが必要 |
| Zip | `dist/*.zip`, 一時フォルダ | 配布用 zip 更新 | 既存 zip の誤上書き | `dist/` 退避、ファイル単位差し戻し |
| MEGA Upload | `data/posts.csv` の `zip_url` | MEGA 上のファイル作成/削除/再作成 | 同名上書き、リンク更新 | CSV 戻し + MEGA 上の確認 |
| Pixiv | ブラウザ profile 内セッション | 投稿画面入力、予約設定 | 誤入力、誤予約 | 送信前に人が画面で中断/修正 |
| Patreon | ブラウザ profile 内セッション | 投稿画面入力 | 誤入力、誤投稿 | 送信前に人が画面で中断/修正 |
| Discord | なし | Webhook 送信 | 重複通知 | 再送防止設計が必要、現状は手動運用 |
| ブラウザ profile 利用 | `browser_profile/` | ログイン状態保持 | ロック、破損、誤初期化 | profile を触らない、必要時は別名退避 |

### 初回リメイク時の禁止事項

- `browser_profile/` の初期化、移行、削除を UI 改修と同時に行わない
- `data/posts.csv` の列削除や列名変更を先行しない
- `Clean` の上書き仕様を、代替退避策なしに変えない
- Discord の再送制御を入れる前に、既存送信フローを勝手に省略しない

## 4. ブラウザ手動確認フロー / 終了責務整理表

| 場面 | 現行の流れ | 誰が確認するか | 終了責務 | keep 条件 |
| --- | --- | --- | --- | --- |
| Pixiv 起動 | shared browser context を起動し、各タスクを新規タブで準備 | ユーザー | 投稿内容を確認し、投稿ボタンを自分で押す | 自動押下しない |
| Patreon 起動 | browser context 上で各タスクの作成画面を準備 | ユーザー | 投稿内容を確認し、公開/投稿を自分で行う | 自動押下しない |
| ログイン必要時 | 投稿画面遷移後、ログインが必要ならそのタブで人が対応 | ユーザー | ログイン完了後に画面確認 | 自動ログイン強制を増やさない |
| ブラウザ profile ロック | Chrome が開いている/ロック中なら retry 案内を出す | ユーザー | 既存ウィンドウを閉じて再試行する | 再試行導線を消さない |
| Web から Pixiv/Patreon 実行 | 新しいコンソールで `manager.py --pixiv/--patreon` を起動 | ユーザー | 別ウィンドウ側を操作 | サブプロセス分離を維持 |
| Web 側の完了表示 | process status をポーリングして completed/terminated を表示 | UI | 状態表示のみ | UI 完了と実際の投稿確定を混同しない |
| CLI 終了時 | `Browsers are kept open` と表示後、Enter でブラウザ close へ進む | ユーザー | 確認が終わってから Enter | ブラウザ即時 close に変えない |
| uploader 単体実行 | `run()` はブラウザを返し、単体 `main()` では Enter 待ち後に close | ユーザー | 確認完了後に閉じる | `run()` と `main()` の責務差を維持 |

### UI 改修時に必ず維持する文脈

- 「処理完了」は「自動入力が終わった」の意味であり、「投稿確定」ではない
- ユーザーが次に取る行動を明示する
- ブラウザを閉じる責務を API 側で早取りしない
