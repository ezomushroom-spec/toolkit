# UI設計原則

## 目的

- 初心者が迷わず主操作に到達できることを最優先にする。
- 既存機能は残しつつ、見つけやすさと安全性を上げる。
- 派手さより、理解のしやすさと失敗しにくさを優先する。

## 改善順序

1. 構造を整える。
2. 見た目を整える。
3. 安全性を整える。

この順序を崩さない。見た目だけ先に触ると、操作導線の問題が残る。
この文書は UI 判断の背景説明であり、案件別の細則は `projects/<name>/AGENTS.md` に置く。

## 構造の原則

- 1画面1目的を意識し、主操作をひとつ決める。
- 入力は意味ごとにまとめ、依存関係がある項目は近くに置く。
- 頻出操作は上部または右下など、探しやすい場所に固定する。
- 補助操作は主操作より目立たせない。
- 専門用語より、動作が想像できる短い日本語を使う。
- 空状態では「何もない」だけで終わらせず、次に何をすればよいか示す。

## 見た目の原則

- 強調は色数より優先順位で作る。主操作、状態、警告の役割を混ぜない。
- 余白でグルーピングし、線や装飾は必要最小限にする。
- ラベル、ボタン、メッセージの文体をそろえる。
- 同じ意味のUIは同じ見た目と配置ルールでそろえる。
- 文字サイズ差は役割差に対応させ、飾り目的で増やさない。
- 一覧と詳細は視線移動が少ない配置を選ぶ。

## 安全性の原則

- 削除、上書き、送信、外部反映は危険操作として扱う。
- 危険操作の前に、対象名、影響範囲、戻せるかどうかを明示する。
- 入力エラーは項目の近くに出し、修正方法を短く書く。
- 通信中や保存中は連打を防ぎ、進行中であることを見せる。
- 例外時は「失敗しました」で終わらせず、再試行可否と確認点を出す。
- エラー表示には、ユーザーが次に取るべき行動を必ず含める。

## 判断基準

- 新しいUI案が既存機能の到達性を下げるなら採用しない。
- 迷うなら、操作数が少ない案より理解しやすい案を選ぶ。
- 変更後は、初見ユーザーが主操作、キャンセル、危険操作を区別できるか確認する。

## この文書に残すもの

- UI をどう判断するかという背景と原則
- 主操作、空状態、危険操作、エラー表示の考え方
- 案件をまたいで再利用できる共通の見方

## この文書に書かないもの

- 特定案件の画面名やライブラリ前提
- 実装担当の作業分担
- サブエージェント向けのプロンプト全文
