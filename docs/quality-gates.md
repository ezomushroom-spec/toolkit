# 品質ゲート

この文書は、Codex が「動いたように見えるだけ」で完了扱いしないための workspace 共通の品質基準をまとめる。
`AGENTS.md` と `docs/workflow.md` が方針と進め方を扱い、この文書が完了判定と報告形式の正本を扱う。
実装詳細や project 固有の build/test/run 手順は `projects/<name>/AGENTS.md` に分ける。

## 1. 完了の定義

次を満たしていない変更は、見た目が動いても完了扱いにしない。

- 変更目的と変更範囲を短く説明できる
- 既存挙動を壊していない根拠がある
- 変更内容に対応した確認方法がある
- 未確認事項が明示されている
- 残るリスクが説明されている
- 戻し方または影響の閉じ込め方が説明できる

## 2. 実装前に確認すること

- 何を変えるか
- 何を変えないか
- 関連画面、関連処理、関連データ
- 正本のコード、設定、データ
- 破壊的操作や誤操作リスク
- 失敗時の戻し方

複雑案件では、着手前に `docs/templates/implementation-plan-template.md` を使って計画を残す。

## 3. 実装中の禁止事項

- 無関係なリファクタを混ぜる
- ついでの命名変更や構成変更を広げる
- 未確認の既存機能を削除する
- エラーを握りつぶして成功扱いにする
- テストや確認手段なしで「たぶん大丈夫」で閉じる

## 4. 確認観点

### 4-1. 基本確認

- 正常系で目的の操作が完了するか
- 既存の主操作や到達経路が残っているか
- 変更していない周辺機能に明らかな悪影響がないか

### 4-2. 失敗系

- 空入力
- 必須値不足
- 不正設定
- 存在しないパス
- 権限不足
- 通信失敗または外部依存の失敗
- 途中停止、キャンセル、ウィンドウクローズ後の状態

### 4-3. UI 安全性

- 主操作が見つけやすいか
- 危険操作に確認、対象明示、結果説明があるか
- 実行中に二重実行できないか
- エラー時に次の行動が分かるか
- 空状態や初回状態で誤解しないか
- ファイル破壊や誤送信につながる導線がないか

### 4-4. 資源使用

- 無駄なメモリ使用が増えていないか
- GPU を使う処理で不要な常駐や解放漏れがないか
- 長時間ジョブで無限待機や無限再試行がないか
- バックグラウンド処理が終了条件を持つか

### 4-5. テスト可能性

- 自動テストがあるなら、今回の変更で実行可能か
- 自動テストがないなら、誰でも再現できる手順が残っているか
- 不具合修正では、再発条件を追える確認方法があるか

## 5. 確認結果の残し方

確認は次のいずれかで残す。

- 実行した自動テスト
- 実行できた手動確認手順
- 実行できなかった場合の理由と代替確認

「テストを追加した」だけでは不十分で、「実行できた」「今回は環境上実行できず、ここまで確認した」を区別して書く。

## 6. 完了報告の形式

完了報告には少なくとも次を含める。

### 6-1. 何を変えたか

- 変更の要点
- 影響する画面、処理、データ

### 6-2. 何を確認したか

- 実行したテストや確認手順
- 確認できた正常系
- 確認できた失敗系や安全性

### 6-3. 何が未確認か

- 環境不足で未実施
- 時間や依存の都合で未実施
- 今回のスコープ外

### 6-4. 残るリスク

- 想定外入力
- 実データでの差
- 性能や資源使用
- 既存導線への副作用

## 7. project 個別に上書きしてよいもの

各 project の `AGENTS.md` では次を具体化してよい。

- build / run / test / lint の方法
- この project で特に壊してはいけない導線
- 高リスク操作の一覧
- 優先確認すべき失敗系
- 配布や運用の制約

workspace 共通ルールを削るためではなく、具体化のために使う。
