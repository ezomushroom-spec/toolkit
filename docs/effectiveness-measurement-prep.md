# 実効性計測の前段整理

この文書は、workspace のルール、skill、完了基準が実際に効いているかを、低コストで見られるようにするための準備メモである。

目的は「厳密な監査」ではなく、次を判断できる状態を作ること。

- どの skill が実務で使われているか
- 完了報告の質が落ちていないか
- docs 更新漏れが増えていないか
- どのルールが実際には守られにくいか

この段階では、まだ自動計測や定期レポートは導入しない。
まずは、何を測るか、どこから拾うか、どこまでを低コストの範囲とするかを固定する。

## 1. 方針

- 手作業で破綻する重い記録は入れない
- 最初は「役に立つ最小指標」だけに絞る
- 行動ログそのものではなく、既存の作業成果物から拾えるものを優先する
- ルール違反率を厳密に出すより、弱い箇所を早めに見つけることを優先する
- 数字だけで判断せず、未確認事項や残るリスクの文章も見る

## 2. 最初に測る指標

最初の計測対象は、次の 4 本に絞る。

### 2.1 skill 利用の偏り

見たいこと:

- 基幹 skill が実際に使われているか
- 一部の skill に偏りすぎていないか
- 追加した skill が使われずに放置されていないか

最初の記録単位:

- 1 案件または 1 大きな依頼ごと
- 使った skill 名を 1 行で残す

### 2.2 完了報告の充足度

見たいこと:

- `docs/quality-gates.md` の必須項目が落ちていないか
- 「何を確認したか」「未確認事項」「残るリスク」が薄くなっていないか

最初の記録単位:

- 完了報告ごと
- 次の 4 項目が埋まったかだけを見る
  - 何を変えたか
  - 何を確認したか
  - 何が未確認か
  - 残るリスク

### 2.3 docs 更新漏れ

見たいこと:

- 実装内容は変わったのに入口文書や判断メモが古いまま残っていないか
- `documentation-update-rules.md` の判定が実務で機能しているか

最初の記録単位:

- 中規模以上の変更ごと
- docs 更新要否を見たかどうかだけ残す

### 2.4 詰まりの型

見たいこと:

- どの段階で詰まりやすいか
- 同じ詰まりが何度も出ていないか

最初の分類:

- 正本未確定
- 仕様不足
- 設計分離不足
- 確認項目不足
- 実装後の副作用
- UI / 操作性不足
- docs 更新漏れ

## 3. どこから拾うか

新しい専用ログを増やさず、まず次から拾う。

### 3.1 完了報告

主に見るもの:

- 何を変えたか
- 何を確認したか
- 何が未確認か
- 残るリスク

ここから拾える指標:

- 完了報告の充足度
- 確認不足の傾向
- 残リスクの種類

### 3.2 案件 docs

主に見るもの:

- `projects/<name>/docs/` の計画、採用判断、現状確認

ここから拾える指標:

- docs 更新漏れ
- 実装前計画の有無
- 詰まりの型

### 3.3 会話ログ

主に見るもの:

- skill 使用
- 方針転換
- 実装前診断の有無

ここから拾える指標:

- skill 利用の偏り
- 診断を飛ばしやすい案件の型

## 4. 保管場所

最初の保管場所は、workspace 共通 docs ではなく、運用記録用の project か docs に寄せる。

候補:

- `projects/workspace-operations/docs/`
- または `docs/metrics/`

現時点では、まず `projects/workspace-operations/docs/` を推奨する。

理由:

- 共通ルール本体と運用観測記録を分けやすい
- 後から archive しやすい
- docs の正本と観測ログを混ぜずに済む

## 5. まず作る最小記録物

本実装の前に、必要な記録物は次の 2 つだけでよい。

### 5.1 計測スナップショット

例:

- `projects/workspace-operations/docs/measurement-snapshot-YYYYMM.md`

最低限の記載項目:

- 期間
- 主な案件
- 使われた skill
- 完了報告の抜けやすい項目
- docs 更新漏れの有無
- よく出た詰まり
- 次月に直すべき運用上の弱点

### 5.2 計測ルール

例:

- `projects/workspace-operations/docs/measurement-rules.md`

最低限の記載項目:

- 見る指標
- 拾う元
- 集計頻度
- 今は見ないもの

## 6. 今は見ないもの

最初は次を対象外にする。

- 厳密なルール違反率
- すべての会話の全件分析
- skill ごとの精密な成功率
- 工数や時間の厳密集計
- 自動イベント収集

これらは価値はあるが、最初から入れると維持コストが勝ちやすい。

## 7. 導入順

Phase 3 の実装は次の順で行う。

1. 計測ルールを 1 枚作る
2. 保管場所を作る
3. 月次または区切り単位のスナップショットを手動で 1 回作る
4. 手動でも続くと判断できたら、AI による半自動集計を検討する
5. それでも価値が高いものだけ、自動化対象に広げる

## 8. 今回時点の結論

- Phase 3 はいきなり自動計測へ行かない
- 最初に見るべきは `skill 利用`, `完了報告の充足度`, `docs 更新漏れ`, `詰まりの型` の 4 本
- 収集元は、完了報告、案件 docs、会話ログに限定する
- 記録物は `measurement-rules.md` と月次スナップショットの 2 種で十分
