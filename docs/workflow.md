# 作業フロー

この文書は、workspace 共通ルールを実作業の流れに落とすための運用文書です。
`AGENTS.md` の方針、`docs/quality-gates.md` の完了基準、`docs/templates/implementation-plan-template.md` の計画テンプレートを、着手から完了までどう使うかを整理します。

## 1. 基本姿勢

- AI は実装補助ではなく管理対象として扱う。
- 人は目的、境界、優先順位、完了条件を決める。
- AI は整理、具体化、草案作成、実装、確認支援を担う。
- AI は曖昧でもそれらしい出力を返す前提で、会話だけに頼らず文書へ残す。
- 問題が出たらコードだけを直して閉じず、必要に応じて仕様、設計、確認項目まで戻る。

## 2. 着手前に固めること

実装前に、少なくとも次の5点を短く固定する。

- 今回の目的
- 今回の対象範囲
- 壊してはいけないもの
- 完了条件
- 確認項目

加えて、次も確認する。

- 対象画面、関連処理、関連データ、戻し方
- 正本のコード、設定、データ
- 危険操作や誤操作リスク
- 今回やらないこと

ここが曖昧なまま、実装や大きな設計変更に進まない。

## 3. workspace での開始手順

- 新機能追加前は `/opsx:new` で仕様化の入口を作る。
- コードを書く前は `/opsx:ff` で仕様書を生成し、確認してから `/opsx:apply` を実行する。
- 中規模以上の案件では、`docs/templates/implementation-plan-template.md` 相当の計画を先に出す。
- 中規模以上の案件で作る調査、計画、判断メモは、原則として `projects/<name>/docs/` に残す。
- 新規 UI、技術選定、アーキテクチャ変更、クロスレイヤー設計を伴う案件では、実装前に方式比較を行う。

小規模修正では最小5項目を優先し、計画は必要最小限でよい。

この文書では `/opsx:new` と `/opsx:ff` を、仕様化と実装前確認の運用名として扱う。
環境上の prompt 名が別でも、作業の段階として対応していればこの呼び方を優先する。

## 4. 標準の進め方

基本の流れは次の順にする。

要求整理  
→ 作業計画  
→ 仕様整理  
→ 設計整理  
→ 確認項目整理  
→ 実装  
→ 検証  
→ 反映・記録

上流の文書ほど「何を決めるか」を固定し、下流ほど「どう実現するか」を具体化する。

### 4.1 要求整理

- 目的、対象、やらないこと、壊してはいけないもの、完了条件を固定する。
- 要求の背後にある制約や成功条件を確認する。
- 再構築案件では、守るべきものが既存資産なのか UX なのかを先に確認する。

### 4.2 作業計画

- 最初に触る場所、後回しにする場所、一度に変える範囲を決める。
- 影響を受ける画面、処理、データ、設定を明示する。
- 削除、停止、置換が必要なものも計画に含める。
- 段階的な確認地点と戻し方を入れる。

### 4.3 仕様整理

- 利用者や呼び出し側から見える振る舞いを固定する。
- 入出力、操作手順、正常時、異常時、保存内容、制約、互換性を明示する。
- 実装都合で勝手に振る舞いを増やさない。
- 仕様は AI への明確な指示でもあるため、曖昧語を減らす。

### 4.4 設計整理

- 仕様をどの構造で実現するかを決める。
- 責務の境界、データの流れ、状態の持ち方、エラー処理、依存関係を整理する。
- 必要以上に複雑な層や抽象化を作らない。
- UI と業務ロジックを混ぜすぎない。

### 4.5 確認項目整理

- 実装前に合否基準を固定する。
- 正常系、異常系、空入力、不正設定、存在しないパス、キャンセル、二重実行、保存と再読込、既存互換を含める。
- 危険操作、エラー表示、空状態、資源使用も確認対象に含める。
- 必要に応じて、確認項目を手動確認手順、テスト観点、テスト設計、自動テストへ落とし込む。

## 5. 実装時の原則

- 一度に広く触りすぎない。
- 既存機能を勝手に削除しない。
- 無関係なリファクタ、命名整理、構成変更を混ぜない。
- 変更理由を説明できる形で直す。
- 例外処理を後回しにしない。
- 危険操作、エラー表示、二重実行防止を後回しにしない。
- 不明点を黙って埋めない。
- 仮対応を入れたら明示する。

## 6. サブエージェント活用

- サブエージェントは、ユーザーが委任を明示的に許可した場合にだけ使う。
- 任せるのは探索、比較、計画、監査、レビュー、限定実装、検証など境界が明確な作業に絞る。
- 実装コード、最終判断、削除判断、統合判断はメイン担当が持つ。
- サブの結果は提案として扱い、採用前にメインが整合性を確認する。

## 7. 検証と戻し方

- 検証は「動いた」ではなく「要求どおりか」を確認する。
- `docs/quality-gates.md` の観点に沿って、正常系、失敗系、安全性、資源使用、テスト可能性を確認する。
- 自動テストがあれば実行し、なければ再現可能な手動確認手順を残す。
- 実行できなかった確認は、理由と代替確認を明示する。

問題が出たときは、次の順で原因を見る。

1. 実装ミスか
2. 設計の切り方が悪いか
3. 仕様の定義が足りないか
4. 要求整理の時点で目的が曖昧だったか

コードだけ直しても再発する場合は、上流の文書まで戻して修正する。

## 8. 完了判定

この文書では、完了判定の入口だけを示す。

- 実装後は、変更目的に対して必要な確認が終わっていることを確認する。
- 未確認事項、残るリスク、戻し方の見通しを残す。
- 完了扱いにする前に、`docs/quality-gates.md` の完了定義と報告形式を確認する。

完了判定、確認観点、完了報告の詳細は `docs/quality-gates.md` を正本とする。

## 9. 参照先の使い分け

- workspace 共通ルール: `AGENTS.md`
- 作業の流れと着手条件: `docs/workflow.md`
- 完了基準と確認観点: `docs/quality-gates.md`
- 実装原則: `docs/coding-rules.md`
- UI 判断の背景: `docs/ui-principles.md`
- 中規模以上の計画: `docs/templates/implementation-plan-template.md`
- 案件固有の制約: `projects/<name>/AGENTS.md`
