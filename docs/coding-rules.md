# 実装ルール

## 基本方針

- 変更は小さく分け、画面、状態管理、データ処理を同時に大改造しない。
- 実装前に影響範囲を整理し、変更理由を一言で説明できる状態にする。
- 不明点があっても挙動を推測で変えず、現状維持を優先する。
- 初心者ユーザーを前提に、理解しやすさと修正しやすさを優先する。
- 実装とファイル更新はメイン担当が行い、サブエージェントの提案は参考情報として扱う。

## UI改修ルール

- 先にレイアウトと導線を整え、その後で配色や装飾を調整する。
- 危険操作は通常操作と見た目を分ける。
- 同じ役割のUIとスタイルは画面をまたいで統一する。
- ボタン文言は動詞で終える。「保存」「削除する」など動作が分かる形にする。
- 主操作は補助操作より目立たせ、配置を安定させる。
- ローディング中は二重実行を防ぎ、待機理由が分かる表示を出す。
- エラーや警告は黙って握りつぶさず、ユーザー向け文言に変換して表示する。
- エラー表示には、再試行、入力修正、設定確認など次の行動を含める。

## コード設計ルール

- 1ファイル1責務を意識し、UI表示、入力検証、外部I/Oを分離する。
- 既存関数を残したまま新旧分岐を増やしすぎない。置換時は呼び出し元を整理する。
- マジックナンバーや文言を散らさず、意味のある名前に寄せる。
- 失敗しうる処理にはタイムアウト、例外処理、戻り値確認を入れる。
- 破壊的処理は実行前に条件確認し、ログまたは通知で結果を追えるようにする。
- 大きな構造変更は、目的、影響範囲、代替案を短く説明できる形で進める。
- 対象画面だけでなく、関連処理、データ影響、戻し方まで見てから変更する。

## テストと確認

- 変更した主操作には最低1つの確認手段を持つ。自動テスト、手順確認、画面確認のいずれでもよい。
- 不具合修正では再発条件を残す。可能ならテスト、難しければ検証手順を記録する。
- UI変更時は正常系だけでなく、空状態、入力エラー、キャンセルも確認する。
- 危険操作やローディング処理を触った場合は、確認導線と二重実行防止を必ず確認する。

## 禁止事項

- 使われている機能を未確認のまま消すこと。
- 例外を無視して成功扱いにすること。
- 初心者が理解できない略語や内部用語をそのまま画面に出すこと。
- サブエージェントに実装を任せて、そのまま採用すること。

## この文書の役割

- メイン担当が実装時に守る設計、例外処理、確認の原則を扱う。
- 監査用の出力形式やプロンプトは `.agents/skills` に分ける。
- 案件固有のライブラリ構成やテスト優先順位は `projects/<name>/AGENTS.md` に分ける。
